# 业务变更 Spec 定义与模板

> 状态：初稿  
> 目的：基于横纵业务演进方法论，定义一份业务 Spec 应该表达什么、如何阅读，以及如何在需求推进过程中持续完善。

## 1. Spec 的定义

在这套方法论中，Spec 不是一份孤立的需求文档，也不只是页面、接口和研发任务的集合。

Spec 是对一次业务变化的完整、可追踪表达：

```text
Spec = 业务变更单元 + 对当前业务模型的引用 + 正确性证据
```

它需要回答：

1. 为什么发生变化；
2. 用户或业务能够观察到什么变化；
3. 完整业务场景如何横向闭环；
4. 哪些领域能力、规则和契约受到影响；
5. 系统如何从旧状态安全演进到新状态；
6. 如何证明变化后的业务是正确的。

Spec 不复制完整的领域、能力、场景和契约定义，而是记录本次变化，并通过稳定标识引用当前有效的业务模型。

## 2. Spec 与当前业务模型的关系

业务知识需要分成两类。

### 2.1 当前业务模型

当前业务模型描述当前有效的业务状态，包括：

- 领域及其责任边界；
- 领域能力；
- 业务概念、状态和规则；
- 领域不变量；
- 跨领域契约；
- 当前有效的业务场景。

它回答：

> 当前业务如何运行，为什么能够这样运行？

### 2.2 业务变更 Spec

业务变更 Spec 描述业务从当前状态演进到目标状态的过程，包括：

- 业务目标；
- Before / After；
- 新增或修改的业务场景；
- 领域、能力、规则和契约变化；
- 数据、兼容、灰度、降级和回滚策略；
- 验收方式及证据。

它回答：

> 为什么要改变、具体改变什么、如何安全地改变，以及如何证明变化正确？

两者应保持以下关系：

```text
当前业务模型
    ↓
业务变更 Spec
    ↓
设计、开发与验证
    ↓
新的当前业务模型
```

因此，不应为每个版本复制一套完整业务模型，而应该维护：

```text
当前有效模型 + 每次变化的 Spec
```

## 3. 如何阅读一份 Spec

阅读一份 Spec 时，应依次回答以下六个问题。

### 3.1 为什么变

- 要解决什么业务问题；
- 业务目标是什么；
- 通过什么业务结果或指标判断成功。

### 3.2 业务怎么变

- 当前业务状态是什么；
- 目标业务状态是什么；
- 用户、运营人员或外部系统能够观察到什么不同。

### 3.3 横向是否闭环

- 场景从什么事件开始；
- 依次经过哪些领域能力；
- 领域之间通过什么契约交接；
- 主路径、异常路径和降级路径如何结束；
- 最终业务结果是否明确。

### 3.4 纵向影响什么

- 复用、扩展、修改、新建、绕过或废弃了哪些能力；
- 哪些领域规则、状态和数据发生变化；
- 是否破坏历史业务语义；
- 哪些已有场景需要回归。

### 3.5 演进是否安全

- 存量数据和进行中状态如何处理；
- 新旧客户端、服务和配置如何兼容；
- 如何灰度、降级和回滚；
- 关闭新入口后，已经产生的订单、任务或权益如何继续处理。

### 3.6 如何证明正确

- 领域不变量由什么测试保证；
- 契约语义和异常协议如何验证；
- 端到端场景如何验收；
- 灰度、迁移和回滚如何演练；
- 生产异常如何发现、对账和补偿；
- 业务目标最终是否达到。

如果一份 Spec 只能回答“需要开发哪些页面、接口和任务”，它仍然是一份实现说明，还没有完整进入业务模型层。

## 4. Spec 的标准结构

### 4.1 基本信息

```text
变更 ID：CHG-<业务标识>
变更名称：
状态：草稿 / 评审中 / 开发中 / 灰度中 / 已完成 / 已废弃
所属项目或版本：REL-<版本标识>
场景负责人：
涉及领域：
关联文档：
```

### 4.2 为什么做

需要表达：

- 业务背景；
- 当前问题；
- 业务目标；
- 非目标；
- 成功指标；
- 关键约束。

这一部分应该能够解释为什么值得改变当前业务模型。

### 4.3 Before / After

使用业务语言描述变化前后的可观察状态。

```text
Before：
当前用户失败后只能结束挑战。

After：
符合条件的用户可以购买一次复活机会，并恢复到约定的挑战状态。
```

Before / After 不应只描述技术实现，例如“新增一个接口”或“增加一张数据表”。

### 4.4 横向业务场景

至少表达：

- 场景 ID；
- 触发条件；
- 参与者；
- 主路径；
- 关键异常与降级路径；
- 最终业务结果；
- 端到端业务标识；
- 观测指标。

业务场景以一次完整、可观察的业务结果为边界。

### 4.5 纵向领域影响

使用影响矩阵表达本次变化对领域能力的影响。

| 领域 | 能力、规则或状态 | 变化类型 | 具体变化 | 历史影响 | 负责人 |
| --- | --- | --- | --- | --- | --- |
| 示例领域 | CAP-EXAMPLE | 复用 | 直接使用已有能力 | 无 | 领域负责人 |
| 示例领域 | INV-EXAMPLE | 修改 | 修改已有业务规则 | 需要回归历史场景 | 领域负责人 |

变化类型统一使用：

| 类型 | 含义 |
| --- | --- |
| 复用 | 使用已有能力，不改变其业务语义 |
| 扩展 | 在原有模型上增加兼容性能力 |
| 修改 | 改变已有规则、状态或业务语义 |
| 新建 | 当前模型无法合理承载，需要建立新能力或领域 |
| 绕过 | 临时绕开已有正式能力或规则 |
| 废弃 | 下线旧能力、规则或契约 |

### 4.6 契约变化

每项新增或修改的契约至少表达：

```text
契约 ID：CTR-<业务标识>
提供方：
使用方：
业务语义：
输入：
输出：
前置条件：
后置条件：
一致性和幂等要求：
失败方式：
重试或补偿方式：
兼容范围：
双方负责人：
```

契约描述的是领域交界处的稳定业务语义，不应退化成接口字段列表。

### 4.7 不变量

不变量分为领域不变量和端到端不变量。

领域不变量由单个领域保证，例如：

```text
一次性权益不能被重复消费。
订单支付成功后不能再次支付。
业务状态只能沿合法路径迁移。
```

端到端不变量属于横向业务场景，例如：

```text
一次成功支付最终对应一次有效权益。
任何一次业务结果都能够追溯到合法来源。
退款、权益与最终业务状态之间始终符合约定。
```

每条端到端不变量都需要拆分为领域责任、契约责任和场景责任。

### 4.8 数据与时间演进

需要说明：

- 是否新增或修改数据语义；
- 历史数据是否迁移；
- 进行中的业务状态如何处理；
- 新旧版本的兼容窗口；
- 灰度范围和扩大条件；
- 功能开关及降级行为；
- 回滚触发条件；
- 回滚后已经产生的业务事实如何继续处理；
- 旧逻辑何时以及如何废弃。

### 4.9 验证矩阵

| 正确性层次 | 验证对象 | 验证方式 | 预期结果 | 证据或链接 | 负责人 |
| --- | --- | --- | --- | --- | --- |
| 领域正确性 | 领域规则、状态机 | 领域回归 | 领域内部保持自洽 |  |  |
| 契约正确性 | 业务语义、幂等、异常协议 | 契约测试 | 领域双方理解一致 |  |  |
| 场景正确性 | 主路径、异常和补偿路径 | 端到端验收、对账 | 完整业务结果成立 |  |  |
| 演进正确性 | 兼容、灰度、迁移、回滚 | 演练和兼容测试 | 新旧状态安全演进 |  |  |
| 观测正确性 | 指标、告警、链路和补偿 | 生产验证 | 异常可发现、定位和恢复 |  |  |

### 4.10 责任与决策

需要明确：

- 场景负责人；
- 各领域负责人；
- 契约双方负责人；
- 重要方案取舍；
- 未决问题；
- 临时方案及退出条件。

如果使用“绕过”或临时方案，还必须记录：

```text
为什么采用临时方案：
绕过了什么正式能力或规则：
产生的业务和系统风险：
影响的业务场景：
后续负责人：
退出条件：
```

## 6. Spec 在研发过程中的形成方式

Spec 不是在需求开始时一次性写完，而是随着业务推进逐步达到可交付状态。

### 6.1 需求提出阶段

首先完成：

- 业务目标；
- Before / After；
- 业务场景；
- 成功指标。

### 6.2 方案设计阶段

继续完成：

- 领域影响矩阵；
- 契约变化；
- 领域和端到端不变量；
- 数据、兼容、灰度和回滚方案；
- 重要决策和临时绕过。

### 6.3 开发测试阶段

继续补充：

- 领域回归；
- 契约测试；
- 端到端验收；
- 兼容与回滚演练；
- 对账和补偿验证；
- 对应的验证证据。

### 6.4 发布完成阶段

最终沉淀：

- 灰度和发布结果；
- 业务指标结果；
- 未解决问题和后续事项；
- 更新后的当前业务模型；
- 本次变化对应的版本记录。

## 7. Spec 完成检查表

一份 Spec 进入研发前，至少应满足：

- [ ] 业务目标和成功指标明确；
- [ ] Before / After 使用业务语言表达；
- [ ] 场景拥有明确触发和可观察结果；
- [ ] 主路径以及关键异常、补偿和降级路径完整；
- [ ] 涉及的领域和能力已经识别；
- [ ] 每项领域变化已经分类；
- [ ] 新增或修改的契约语义明确；
- [ ] 领域不变量和端到端不变量明确；
- [ ] 存量数据和进行中状态有处理方案；
- [ ] 兼容、灰度、降级和回滚方案明确；
- [ ] 可以从变更推导出领域、契约、场景、演进和观测验证；
- [ ] 横向、纵向和契约责任人明确；
- [ ] 临时方案具有负责人和退出条件；
- [ ] 所有关联业务对象使用稳定标识引用，而不是重复复制。

一份 Spec 完成发布后，还应满足：

- [ ] 验证矩阵包含实际证据；
- [ ] 业务指标已经观察并记录；
- [ ] 当前业务模型已经更新；
- [ ] 版本变化记录已经归档；
- [ ] 临时绕过和后续事项已经进入持续跟踪。

## 8. 最终判断

基于横纵业务演进方法论，一份合格的 Spec 应该形成下面这条可追踪链：

```text
业务目标
  → Before / After
  → 横向业务场景
  → 纵向领域能力
  → 跨领域契约
  → 领域与端到端不变量
  → 数据和时间演进
  → 验证矩阵与实际证据
```

Spec 的价值不在于增加一份文档，而在于让参与者能够共同回答：

> 为什么变化、改变什么、影响哪里、如何安全交付，以及如何证明业务变化是正确的？
