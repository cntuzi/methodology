# 横纵业务演进方法论

> 状态：初稿  
> 目的：建立一套同时表达跨领域业务协作、领域能力沉淀和版本演进的可维护方法。

## 1. 问题背景

在业务和产品研发中，同一个需求通常存在两个观察方向。

横向看，一个业务目标往往需要串联多个业务域。例如一次版本交付，可能同时涉及可玩功能、商业化、增长、安全、国际化和运营。单个功能分别完成并不意味着业务目标成立，只有这些功能在同一个业务场景中正确连接，业务才能形成闭环。

纵向看，一个业务域并不是为当前版本临时存在的功能集合。它需要拥有自己的概念、规则、状态和运行逻辑，在满足当前需求的同时，兼容历史逻辑并为后续需求持续沉淀能力。

因此，产品研发不能只回答“这个版本做了哪些功能”，还需要持续回答：

1. 当前业务目标如何跨领域成立？
2. 每个领域为此复用、扩展或新增了什么能力？
3. 本次变化是否破坏了历史业务规则和已有场景？
4. 不同领域、不同版本在灰度、迁移和回滚过程中能否正确协作？

## 2. 核心判断

这套思维方式可以概括为：

> 业务通过横向场景产生价值，通过纵向领域沉淀能力，并在时间维度上持续演进。

其中：

- 横向负责“这次业务目标能否成立”。
- 纵向负责“长期能力能否持续演进”。
- 时间负责“领域能力和领域连接如何从一个状态变化到另一个状态”。

横向和纵向不是两套独立系统。一次需求既会改变横向业务场景，也可能改变纵向领域能力以及领域之间的契约。

## 3. 从二维视角扩展为三维模型

最初可以把横向和纵向理解为一个二维平面：

- 横向：多个领域之间的业务连接。
- 纵向：单个领域内部的能力深度。

但时间不宜完全归入横向。更准确的模型包含三个维度。

### 3.1 横向：业务场景与价值链

横向表达一个业务目标如何通过多个领域协作形成闭环，例如：

```text
业务目标：提升新用户付费转化
  → 可玩内容
  → 商业化
  → 增长
  → 安全
  → 国际化
```

横向不是简单的功能清单，也不一定是一条直线。真实业务通常存在分支、依赖、异常和补偿，更接近一张场景关系图。

### 3.2 纵向：领域能力与内部规则

纵向表达一个领域内部相对稳定的业务结构，包括：

- 核心业务概念；
- 能力和职责边界；
- 业务规则与不变量；
- 状态及状态转换；
- 数据所有权；
- 对外契约；
- 历史逻辑与回归要求。

纵向沉淀的不是功能的简单累加，而是能够被不同业务场景复用和组合的领域能力。

### 3.3 时间：版本与演进

时间表示业务系统如何从一个稳定状态演进到另一个稳定状态。

一个版本可以理解为系统在某个时间点的切片，也可以理解为对业务模型的一组变更：

- 新增或修改了哪些场景；
- 复用、扩展或新增了哪些能力；
- 新增、修改或废弃了哪些领域契约；
- 哪些规则、数据和状态需要迁移；
- 新旧版本如何兼容；
- 如何灰度、降级和回滚。

## 4. 底层表达：一张随时间演进的业务关系图

不应分别维护互不关联的横向文档和纵向文档。更合适的方式是维护一组统一的业务对象及其关系，然后从同一份事实生成不同视图。

底层模型包含：

- 节点：领域、能力、业务状态；
- 连线：领域之间的契约和依赖；
- 场景：为了实现业务目标，在关系图上经过的一条或多条路径；
- 版本：特定时间对节点、连线和场景产生的一组变化。

关系可以简化为：

```text
业务场景引用领域能力
领域能力通过契约协作
版本修改场景、能力和契约
领域规则产生纵向回归
业务场景产生横向验收
版本变化产生演进与兼容验证
```

## 5. 四类核心业务对象

### 5.1 领域 Domain

领域是相对稳定的业务责任边界，例如可玩内容、商业化、增长、安全、国际化和运营。

领域需要回答：

- 解决什么业务问题；
- 拥有哪些核心业务事实和数据；
- 包含哪些核心概念；
- 有哪些不可破坏的规则；
- 对外提供哪些能力；
- 明确不负责什么；
- 由谁长期维护。

领域不应该随着每个版本频繁建立和撤销。

### 5.2 能力 Capability

能力是领域内部沉淀、可以被业务场景复用的业务单元。

例如商业化领域可以逐渐形成以下能力结构：

```text
商业化
├── 商品定义
├── 用户购买资格
├── 定价与优惠
├── 订单创建
├── 支付
├── 权益履约
└── 退款
```

这比按照“首充、订阅、礼包、折扣”等具体功能堆积更稳定。具体业务功能通常是多个底层能力的组合。

一个能力至少应记录：

```text
能力 ID：CAP-PURCHASE-ELIGIBILITY
能力名称：购买资格判断
所属领域：商业化
职责：判断指定用户在特定上下文中能否购买商品
输入：用户、商品、业务上下文
输出：允许，或拒绝及原因
核心规则：同一限购周期内只能购买一次
对外契约：checkPurchaseEligibility
负责人：领域负责人
回归测试：REG-COM-012
```

### 5.3 场景 Scenario

场景表达完整的横向业务目标，应该按照业务闭环而不是系统模块组织。

一个场景至少应记录：

- 业务目标；
- 触发条件；
- 参与者；
- 主路径；
- 异常和降级路径；
- 参与的领域和能力；
- 跨领域契约；
- 最终业务结果；
- 观测指标；
- 横向验收用例。

### 5.4 契约 Contract

契约是横向场景和纵向能力真正发生交叉的位置。

契约不仅是技术接口，还需要表达：

- 提供方和使用方；
- 输入和输出；
- 业务语义；
- 前置和后置条件；
- 一致性及幂等要求；
- 失败方式和补偿方式；
- 版本兼容范围；
- 双方责任边界。

示例：

```text
契约 ID：CTR-REVIVE-ENTITLEMENT
契约名称：复活权益发放
提供方：商业化
使用方：可玩内容
前置条件：订单支付成功
输出：一次性复活权益
业务语义：同一权益最多消费一次
失败处理：超时允许重试，重复请求必须幂等
兼容要求：支持约定兼容窗口内的客户端版本
```

## 6. 三种核心视图

底层业务对象保持唯一，但可以根据不同问题生成三种视图。

### 6.1 纵向视图：领域能力地图

领域能力地图用于回答：

- 领域当前具备哪些能力；
- 内部概念、规则和状态如何组织；
- 对外提供哪些契约；
- 本次需求是在复用、扩展还是破坏已有能力；
- 历史逻辑由哪些回归用例保证。

表达结构：

```text
领域
  → 核心概念
  → 领域能力
  → 规则与不变量
  → 对外契约
  → 领域回归测试
```

### 6.2 横向视图：业务场景图

业务场景图用于回答：

- 当前业务目标如何跨领域闭环；
- 场景经过了哪些领域能力；
- 领域之间在哪里交接；
- 正常、异常和降级情况下分别如何处理；
- 最终业务结果如何验证。

表达结构：

```text
业务目标
  → 用户或事件触发
  → 领域能力 A
  → 跨领域契约 A-B
  → 领域能力 B
  → 跨领域契约 B-C
  → 最终业务结果
```

### 6.3 时间视图：版本变更集

版本不只是功能列表，而是业务模型在一个时间段内的变化集合。

| 变更类型 | 需要表达的内容 |
| --- | --- |
| 场景变化 | 新增、修改或下线了哪些业务场景 |
| 能力变化 | 复用、扩展、新增或废弃了哪些能力 |
| 契约变化 | 新增或修改了哪些领域连接 |
| 规则变化 | 哪些领域规则和不变量发生变化 |
| 数据变化 | 是否涉及历史数据迁移和数据兼容 |
| 发布变化 | 灰度、开关、降级和回滚如何处理 |
| 兼容变化 | 支持哪些新旧客户端、服务或业务状态组合 |

## 7. 示例：玩家失败后付费复活

### 7.1 横向业务场景

```text
场景 ID：SCN-PAID-REVIVE
场景名称：玩家失败后付费复活
业务目标：提升关卡完成率和付费转化
触发条件：玩家挑战失败
参与领域：可玩内容、商业化、安全、增长

主路径：
1. 可玩内容产生“挑战失败”状态。
2. 商业化判断用户是否具有复活购买资格。
3. 商业化创建订单并完成支付。
4. 商业化发放复活权益。
5. 可玩内容消费权益并恢复挑战状态。
6. 增长记录曝光、点击、支付和复活结果。

业务结果：
- 玩家恢复到指定游戏状态；
- 复活权益只能消费一次；
- 支付、权益和游戏状态可以对账。
```

### 7.2 纵向领域变化

可玩内容领域：

- 扩展挑战状态机；
- 增加从“失败待复活”恢复到“挑战中”的合法路径；
- 保证非法或重复复活不会改变游戏状态。

商业化领域：

- 复用购买资格、订单和支付能力；
- 扩展权益履约，支持一次性消费权益；
- 保证重复通知不会重复发放权益。

增长领域：

- 复用实验曝光和行为归因能力；
- 增加复活场景的业务事件语义。

### 7.3 跨领域契约变化

- 可玩内容向商业化提供挑战和用户上下文；
- 商业化向可玩内容提供复活权益；
- 可玩内容向增长提供复活结果；
- 支付、权益消费和游戏状态之间需要可追踪和可对账。

### 7.4 版本变更集

```text
版本 ID：REL-2026-08
新增场景：SCN-PAID-REVIVE
扩展能力：CAP-ENTITLEMENT、CAP-GAME-STATE-RECOVERY
新增契约：CTR-REVIVE-ENTITLEMENT
新增规则：INV-ENTITLEMENT-CONSUME-ONCE
兼容处理：不支持该能力的客户端不展示复活入口
回滚策略：关闭购买入口，但不回滚已经支付和发放的权益
```

## 8. 验收体系

每次版本验收由纵向领域回归、横向场景验收和演进兼容验收共同组成。

### 8.1 纵向领域回归

纵向回归来源于领域规则、不变量和状态机，例如：

- 已支付订单不能再次支付；
- 一次支付不能重复发放权益；
- 一次性权益不能消费两次；
- 游戏状态只能按照合法路径迁移；
- 新增复活能力后，历史非复活玩法仍然正确。

### 8.2 横向场景验收

横向验收来源于业务场景，例如：

- 正常完成一次付费复活；
- 支付成功但权益发放超时；
- 权益发放成功但游戏状态恢复失败；
- 用户重复点击购买；
- 不同地区、币种、语言和风控结果下流程正确；
- 各领域数据最终可以对账。

### 8.3 演进与兼容验收

演进验收来源于版本变化，例如：

- 新服务与旧客户端组合；
- 灰度用户与非灰度用户同时存在；
- 新旧配置同时生效；
- 功能关闭、服务降级和版本回滚；
- 历史订单、历史权益和进行中的游戏状态如何处理。

版本准入条件可以概括为：

```text
版本可发布
= 受影响领域回归通过
+ 关键业务场景通过
+ 变更契约兼容验证通过
+ 灰度、降级和回滚路径通过
```

不应无限期保证所有历史版本的任意组合都能够互相连接。需要明确兼容窗口，包括支持的客户端版本、服务版本、历史数据和旧业务流程，并对超出窗口的状态制定升级或下线策略。

## 9. 可维护性设计

### 9.1 只维护一份业务事实

不要分别维护相互复制的领域文档、版本文档和测试文档。每个业务对象拥有稳定标识，其他视图通过引用建立联系。

```text
领域：DOM-COMMERCE
能力：CAP-PURCHASE-ELIGIBILITY
场景：SCN-PAID-REVIVE
契约：CTR-REVIVE-ENTITLEMENT
规则：INV-ENTITLEMENT-CONSUME-ONCE
版本：REL-2026-08
测试：REG-COM-012 / E2E-REVIVE-003
```

版本文档只记录引用和变化：

```text
REL-2026-08
新增：SCN-PAID-REVIVE
扩展：CAP-ENTITLEMENT
新增：CTR-REVIVE-ENTITLEMENT
影响：INV-ENTITLEMENT-CONSUME-ONCE
验收：E2E-REVIVE-001 ～ E2E-REVIVE-008
```

由此可以查询：

- 某个版本改变了什么；
- 某个领域长期积累了什么；
- 某个场景依赖哪些能力；
- 某个能力被哪些场景使用；
- 修改一个契约会影响哪些业务；
- 哪些测试需要回归。

### 9.2 控制表达粒度

为了避免文档系统快速膨胀，应优先记录：

- 跨领域协作；
- 稳定且重要的业务概念；
- 容易被破坏的业务不变量；
- 状态和数据所有权；
- 高风险异常及补偿路径；
- 对后续需求有复用价值的领域能力。

不需要把每个页面、按钮、代码调用或临时实现细节都提升为领域能力。

### 9.3 保持当前状态和演进记录分离

- 领域、能力和契约文档描述当前有效状态；
- 版本文档描述从旧状态到新状态的变化；
- 重要取舍通过决策记录保留原因；
- 历史状态由版本控制系统追溯，不在当前文档中重复堆积。

## 10. 责任机制

这套体系需要三种责任角色。

### 10.1 领域负责人

负责保证领域内部概念、规则、能力、数据和状态自洽，并维护领域回归范围。

### 10.2 场景负责人

负责保证业务目标能够横向闭环，关注最终业务结果，而不是只关注参与方是否分别完成开发。

### 10.3 契约双方负责人

提供方和使用方共同确认契约的业务语义、异常处理、兼容范围和演进方式。

每个需求进入研发前，至少回答：

1. 横向上新增或修改了哪些业务场景？
2. 纵向上复用、扩展、新增或破坏了哪些领域能力？
3. 修改了哪些跨领域契约？
4. 哪些领域规则和历史场景需要回归？
5. 发布过程中的灰度、兼容、降级和回滚如何处理？

如果一个需求只能回答“需要开发哪些页面、接口和任务”，说明它还没有进入业务模型层。

## 11. 建议的文件组织

初期可以使用 Markdown、Mermaid 和 Git 管理，不必立即建设复杂平台。

```text
business-model/
├── README.md
├── glossary.md
├── domains/
│   ├── gameplay.md
│   └── commerce.md
├── capabilities/
│   ├── purchase-eligibility.md
│   └── entitlement.md
├── scenarios/
│   └── paid-revive.md
├── contracts/
│   └── revive-entitlement.md
├── releases/
│   └── 2026-08.md
└── decisions/
    └── entitlement-consistency.md
```

其中：

- `glossary.md` 统一跨团队业务语言；
- `domains/` 记录领域边界和领域全貌；
- `capabilities/` 记录可复用能力、规则和测试；
- `scenarios/` 记录端到端业务闭环；
- `contracts/` 记录领域连接处的稳定语义；
- `releases/` 记录特定时间段的业务模型变化；
- `decisions/` 记录重要设计取舍及其背景。

## 12. 最小可行落地路径

不应从完整建模整个公司开始。可以选择一个正在推进的真实版本，用较小范围验证模型是否有效。

第一阶段只建立：

1. 五到八个一级领域；
2. 三到五条关键业务场景；
3. 场景涉及的核心领域能力；
4. 重要的跨领域契约；
5. 关键领域不变量及对应测试；
6. 当前版本的业务模型变更集。

完成后，用以下问题检验表达体系是否有效：

- 能否清楚解释当前版本的完整业务闭环？
- 能否识别单个需求对其他领域和历史场景的影响？
- 能否从领域视角看到能力的复用和积累？
- 能否据此确定横向验收、纵向回归和兼容验证范围？
- 当人员变化时，新成员能否理解业务为什么这样运行？

如果这些问题能够被稳定回答，再逐步扩展覆盖范围和自动化程度。

## 13. 需要持续警惕的偏差

### 13.1 只有横向，没有纵向

每个版本都能快速交付，但领域内部不断增加特例和耦合，历史逻辑越来越难以回归，最终每个新需求都需要重新理解整个系统。

### 13.2 只有纵向，没有横向

各领域内部设计完整，但没人对最终业务闭环负责，容易出现局部正确、整体不可用的情况。

### 13.3 为未来过度抽象

领域能力应该从重复出现的需求和相对稳定的业务规则中生长，而不是提前设计能够处理所有可能性的万能模型。

### 13.4 用功能模块代替业务领域

页面、接口和技术服务不天然等于业务领域。领域边界应由业务事实、规则、数据所有权和变化原因决定。

### 13.5 只验证最新稳定状态

生产发布会经历新旧版本共存、灰度、迁移、失败和回滚。连接关系不仅要在最终状态正确，也要在演进过程中保持可控。

## 14. 当前方法论总结

横向、纵向和时间不是相互竞争的视角，而是同一个业务系统的不同投影：

- 横向场景保证业务价值能够闭环；
- 纵向领域保证业务能力能够积累；
- 跨领域契约保证不同能力能够稳定连接；
- 版本变更集保证系统能够持续演进；
- 领域回归、场景验收和兼容验证共同保证变化正确。

最终目标不是建立更多文档，而是形成一套能够持续回答“当前业务如何运行、这次变化影响什么、长期能力沉淀在哪里”的共同语言和可追踪模型。
