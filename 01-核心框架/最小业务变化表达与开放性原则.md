# 最小业务变化表达与开放性原则

> 状态：V1 核心结构候选  
> 目的：定义一次业务变化最少需要表达什么，同时确保方法论不被 YAML、表格、固定 Schema 或具体工具限制。

## 1. 核心结论

一次业务变化最少需要回答四类问题：

```text
为什么变化
改变了什么
何时、对谁生效
如何证明变化正确
```

可以概括为：

```text
业务变化
= 变化意图
+ 模型差异
+ 生效条件
+ 正确性承诺
```

这四部分定义的是必须表达的业务语义，而不是一套固定字段，也不是要求使用 YAML、JSON、数据库表或表单进行填写。

同一项业务变化可以根据实际复杂度使用：

- 自然语言；
- Markdown；
- 表格；
- 流程图；
- 状态图；
- 领域模型图；
- 示例和反例；
- 测试、监控、对账和决策记录链接；
- 未来尚未出现的新表达形式。

方法论约束“需要回答什么”，不强制“必须以什么格式回答”。

## 2. 为什么不能被结构化格式限制

业务语义经常具有以下特点：

- 存在上下文和例外；
- 规则之间存在优先级和解释空间；
- 同一个词在不同领域具有不同含义；
- 有些问题适合用状态机表达，有些适合用场景叙述；
- 新业务会产生当前 Schema 无法预见的概念；
- 重要决策通常包含动机、取舍和被放弃的方案；
- 不确定性本身有时也是需要保留的信息。

如果强制将所有业务内容拆成预设字段，会产生两个问题：

1. 为了适配字段而扭曲业务事实；
2. Schema 不断扩展，最终工具结构反过来主导业务思考。

因此，本方法论不追求把全部业务知识转化为机器可计算的数据。

它只要求保留少量能够建立连接和形成视图的结构化锚点，其余业务含义保持开放表达。

## 3. 双层表达模型

为了同时满足可维护性和表达自由，可以把业务模型分成两层。

### 3.1 结构化内核

结构化内核只负责机器必须理解的最小信息：

```text
稳定身份
对象类型
对象关系
变化动作
生命周期状态
生效条件
引用关系
```

例如：

```text
CHG-PAID-REVIVE 是一次业务变化
它修改 CAP-ENTITLEMENT
它新增 SCN-PAID-REVIVE
SCN-PAID-REVIVE 使用 CAP-PAYMENT
它受到 INV-ENTITLEMENT-ONCE 约束
它在特定时间和范围内生效
```

结构化内核的作用是支持：

- 稳定引用；
- 关系查询；
- 影响分析；
- 生效判断；
- 历史追踪；
- 多视图投影。

### 3.2 开放语义正文

开放语义正文负责保存完整业务含义：

- 业务背景；
- 目标和判断；
- 规则解释；
- 正常与异常场景；
- 状态变化；
- 业务取舍；
- 示例和反例；
- 尚未确定的问题；
- 图、表、附件和外部证据。

正文不要求转换为固定字段。工具即使无法理解全部内容，也必须完整保存并允许人阅读。

两层关系可以概括为：

> 结构化内核负责连接，开放语义正文负责表达。

或者进一步表述为：

> 结构约束关系，不限制意义。

## 4. 最小业务变化的四类语义

### 4.1 变化意图

回答：为什么要发生这次变化？

至少应让读者理解：

- 变化的业务背景；
- 想解决的问题；
- 预期产生的业务结果；
- 为什么当前业务状态不能满足要求。

这部分可以是一段判断、一个业务分析、一组数据或一份决策记录，不要求拆成固定表单字段。

如果没有变化意图，记录只能说明“系统改了什么”，不能解释“为什么应该这样改”。

### 4.2 模型差异

回答：这次变化改变了当前业务模型中的什么？

差异可以作用于：

- 领域；
- 能力；
- 契约；
- 场景；
- 不变量；
- 这些对象之间的关系。

从形成新状态的角度，最小变化动作只需要：

```text
新建
修改
失效
```

“复用、扩展、替换、绕过”等词描述变化的业务性质，而不是底层状态操作。

例如：

```text
修改 CAP-ENTITLEMENT
变化性质：扩展
业务含义：增加一次性复活权益

新建 CTR-REVIVE-ENTITLEMENT
变化性质：正式新增

修改 SCN-GAME-FAILURE
变化性质：扩展
业务含义：增加付费复活分支
```

具体“修改了什么”可以通过自然语言、前后对比、状态图或领域模型图表达，不要求压缩成固定 Patch 格式。

### 4.3 生效条件

回答：变化在什么情况下成为实际业务状态的一部分？

至少需要表达：

- 当前生命周期状态；
- 生效时间或生效事件；
- 生效范围；
- 失效时间或退出条件。

生命周期可以使用最小语义：

```text
提议
已确认
已生效
已失效
```

生效范围可以根据业务实际表达：

- 用户或实验分组；
- 国家和地区；
- 租户；
- 客户端或服务版本；
- 功能开关；
- 业务配置；
- 其他领域特有条件。

这里不应该预设一个封闭字段集合。方法论只要求能够回答“何时、对谁有效”。

因此，有效业务模型不是单纯的 `M(t)`，而更接近：

```text
M(时间, 生效上下文)
```

### 4.4 正确性承诺

回答：变化以后，哪些业务事实必须继续成立？

至少需要表达：

- 新增或修改了什么业务不变量；
- 哪些历史不变量受到影响；
- 横向业务场景需要得到什么结果；
- 使用什么证据证明变化正确；
- 当前是否存在验证缺口。

证据可以是：

- 自动化测试；
- 人工验收；
- 业务对账；
- 数据指标；
- 监控和告警；
- 灰度观察；
- 演练记录；
- 决策者确认。

方法论不把“测试用例”视为唯一正确性证据，也不要求将复杂业务承诺压缩成布尔字段。

## 5. 一个不受格式限制的示例

下面的示例使用文本和引用表达，而不是规定某种数据格式。

### CHG-PAID-REVIVE：玩家失败后支持付费复活

#### 为什么变化

部分高意愿玩家在挑战失败后仍有强烈的继续意愿，但当前只能重新开始。希望通过付费复活形成新的继续路径，同时验证其对关卡完成率和付费转化的影响。

预期结果是：符合资格的玩家能够完成一次合法的付费复活，并恢复到约定的游戏状态。

#### 改变了什么

- 扩展 `CAP-GAME-STATE-RECOVERY`，增加付费复活状态恢复能力；
- 扩展 `CAP-ENTITLEMENT`，支持一次性复活权益；
- 新增 `CTR-REVIVE-ENTITLEMENT`，连接商业化与可玩内容；
- 新增 `SCN-PAID-REVIVE`，描述端到端付费复活场景；
- 新增 `INV-PAID-REVIVE-EXACTLY-ONCE`，约束支付、权益与复活结果。

具体的游戏状态变化使用状态图表达；支付、权益和游戏状态之间的异常补偿使用场景分支说明。

#### 何时、对谁生效

变化计划在目标版本发布后首先对实验组生效，仅支持具备该能力的新客户端，并由 `paid_revive_enabled` 开关控制。灰度期间旧业务场景继续有效。完成业务指标和异常率观察后，再决定是否扩大范围。

#### 如何证明正确

需要证明：

- 一次成功支付最多产生一份复活权益；
- 一份权益最多完成一次复活；
- 支付成功但后续失败时可以被发现和补偿；
- 不支持该能力的客户端继续运行旧场景；
- 关闭功能开关后不再产生新订单，但已有订单仍能完成处理。

对应证据可以引用领域回归、端到端验收、支付权益对账和灰度监控。尚未建立证据的承诺应在验证视图中显示为缺口。

这个示例满足四类最小语义，但没有被某一种结构化格式限制。

## 6. 如何从开放表达生成视图

视图生成不要求工具理解所有自然语言，只要求工具能够理解结构化内核中的身份和关系。

### 6.1 横向视图

从场景引用关系出发：

```text
场景
→ 使用的能力
→ 能力之间的契约
→ 端到端不变量
```

工具可以生成关系骨架，场景正文负责解释完整业务含义。

### 6.2 纵向视图

按照领域归属关系聚合：

```text
领域
→ 拥有的能力
→ 领域不变量
→ 对外契约
→ 引用这些能力的场景
```

工具负责整理关联，领域正文负责解释能力为什么这样组织。

### 6.3 版本视图

按照版本或发布时间聚合业务变化：

```text
版本
→ 包含的业务变化
→ 直接修改的业务对象
→ 生效时间和范围
```

版本视图只展示变化，不复制所有业务对象的正文。

### 6.4 验证视图

从变化对象反向查询：

```text
直接变化
→ 相关领域和契约
→ 引用它们的业务场景
→ 相关不变量
→ 已有验证证据或验证缺口
```

工具能够生成验证骨架，但完整的正确性判断仍然由业务语义和实际证据完成。

## 7. 显式变化与推导影响

一次业务变化只显式记录它直接改变了什么。

例如：

```text
直接修改：CAP-ENTITLEMENT
```

哪些场景受到影响，通过既有关系推导：

```text
CAP-ENTITLEMENT
← 被 SCN-PAID-REVIVE 使用
← 被 SCN-SUBSCRIPTION 使用
← 被 SCN-GIFT-PACK 使用
```

因此：

> 直接变化被记录，间接影响被推导。

如果将所有间接影响手工复制到业务变化中，会重新产生重复维护问题。

推导结果也不等于最终判断。工具可以提示三个场景可能受影响，但是否真正需要修改或回归，仍然允许人根据业务语义确认。

## 8. 版本和工具的位置

版本是多个业务变化的交付集合：

```text
版本
→ 包含多个业务变化
```

版本不直接定义领域和业务场景，也不单独决定业务是否已经对所有用户生效。

工具只负责：

- 保存业务对象和开放正文；
- 维护稳定身份和引用；
- 记录对象关系与业务变化；
- 根据时间和范围选择有效变化；
- 查询关系和提示可能影响；
- 生成不同视图的骨架；
- 检查引用缺失、重复定义和失效关系。

工具不负责决定：

- 什么是正确的领域边界；
- 什么构成完整业务场景；
- 一项规则的真实业务含义；
- 某个推导影响是否一定成立；
- 什么证据足以证明业务正确。

## 9. 防止结构反向限制业务的原则

后续无论选择何种载体，都应遵守以下原则：

1. 先表达业务事实，再考虑如何结构化；
2. 结构化只覆盖稳定身份、关系、生效和引用；
3. 所有对象都允许保留开放正文；
4. 允许使用领域特有属性，不建立封闭字段全集；
5. 新内容无法被当前 Schema 理解时，必须允许先保存，而不是拒绝表达；
6. 派生视图展示关系骨架，同时保留返回原始正文的路径；
7. 机器推导只提供候选影响，不替代人的业务判断；
8. Schema 服务于业务模型，业务模型不服务于 Schema。

## 10. 最终收敛

最小业务变化结构不是一个 YAML Schema，而是四类不可缺少的业务语义：

```text
变化意图：为什么变化
模型差异：改变了什么
生效条件：何时、对谁有效
正确性承诺：如何证明正确
```

它与统一业务模型结合后，可以形成新的有效业务状态，并推导横向、纵向、版本和验证视图。

其中：

- 少量结构化信息提供连接和计算能力；
- 开放正文保留完整业务语义和未来扩展性；
- 视图来自同一份事实和关系，而不是复制维护；
- 工具实现可以变化，但四类业务语义和开放性原则保持稳定。

因此，这套方法论追求的不是“把业务完全结构化”，而是：

> 在不损失业务表达自由的前提下，为业务事实建立足以连接、追踪、演进和投影的最小结构。
