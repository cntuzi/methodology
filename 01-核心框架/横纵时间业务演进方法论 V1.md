# 横纵时间业务演进方法论 V1

> 版本：V1  
> 状态：理论结构已收敛，待真实业务案例验证  
> 目的：用一套统一、开放、可追踪的方式描述当前业务、跨领域协作和业务持续变化。

## 0. 一句话定义

> 业务通过横向场景产生价值，通过纵向领域沉淀能力，并通过业务变化在时间中持续演进。

这套方法论不是“如何设计所有业务领域”的标准答案，而是：

> 一套使用统一业务事实，描述业务当前状态、跨领域运行方式、变化过程和正确性证据的方法。

## 1. 要解决的问题

业务产品研发始终面对两个同时存在的要求。

一方面，一个业务目标通常需要串联多个领域。例如一次新玩法交付，可能同时涉及可玩内容、商业化、增长、安全、国际化和运营。每个领域分别完成自己的功能，不代表最终业务能够正确闭环。

另一方面，每个领域不能只为当前版本临时增加功能。它需要维护自己的业务概念、状态、规则和能力，在支持当前业务的同时保持历史逻辑正确，并能够继续演进。

因此，方法论需要持续回答四个问题：

1. 当前业务如何运行？
2. 一个完整业务场景如何跨领域闭环？
3. 本次业务变化修改了哪些能力、规则和连接？
4. 如何证明变化后的业务仍然正确？

## 2. 横向、纵向与时间

### 2.1 横向：业务场景

横向表达一个业务目标如何调用多个领域能力，最终形成可观察的业务结果。

```text
业务目标
→ 场景触发
→ 领域能力 A
→ 领域契约 A-B
→ 领域能力 B
→ 最终业务结果
```

横向不是功能清单，也不一定是一条直线。它可以包含分支、失败、降级、重试和补偿。

横向需要保证的是：

- 场景从触发到结果能够闭环；
- 不同领域对交界处的业务语义理解一致；
- 异常情况下仍有明确结果；
- 端到端业务承诺得到满足。

### 2.2 纵向：领域能力

纵向表达单个领域内部长期稳定的业务结构：

```text
领域
→ 业务事实与状态
→ 业务规则与不变量
→ 对外提供的能力
→ 与其他领域的契约
```

纵向沉淀的不是功能数量，而是可以被不同场景复用和组合的业务能力。

领域是业务事实、规则和状态的责任边界；能力是领域基于这些事实、规则和状态，对外完成的一种业务行为或判断。

### 2.3 时间：业务变化

时间表达业务如何从一个有效状态演进到另一个有效状态。

```text
当前业务模型
+ 业务变化
+ 生效条件
→ 新的有效业务模型
```

版本只是多个业务变化的交付集合。真正改变业务状态的是业务变化以及它的生效条件。

## 3. 核心原则

### 3.1 单一事实，多种视图

横向、纵向、版本和验证不是四份独立数据，而是同一组业务事实的不同投影。

### 3.2 当前状态与变化记录分离

- 当前业务模型描述当前有效的业务；
- 业务变化描述为什么改、改了什么以及如何生效；
- 历史由变化记录保留，不堆积在当前状态正文中。

### 3.3 直接变化被记录，间接影响被推导

一次变化只显式记录它直接改变的对象和关系。受到影响的领域、场景和验证范围，通过已有关系反向推导。

### 3.4 结构约束关系，不限制意义

方法论规定必须表达的业务语义，但不规定必须使用 YAML、JSON、固定表格或封闭 Schema。

### 3.5 先形成最小闭环，再由实践推动演进

V1 不试图覆盖所有领域划分、组织治理、自动化和特殊场景问题。只有真实案例无法表达时，才补充核心模型。

## 4. 当前业务模型

当前业务模型由五类核心业务对象及其关系构成。

### 4.1 领域 Domain

领域是业务责任以及业务事实、状态和规则的所有权边界。

领域需要能够解释：

- 负责解决什么业务问题；
- 拥有哪些核心概念、事实和数据；
- 维护哪些规则和不变量；
- 对外提供哪些能力；
- 明确不负责什么。

### 4.2 能力 Capability

能力是领域可以被场景复用的一种业务行为或判断。

例如商业化领域可能拥有：

```text
商品定义
购买资格判断
定价与优惠
订单创建
支付
权益履约
退款
```

具体功能通常是多个能力的组合，不应简单等同于领域能力。

### 4.3 契约 Contract

契约表达领域交界处的稳定业务语义，包括：

- 提供方和使用方；
- 输入、输出和业务含义；
- 前置和后置条件；
- 一致性与幂等要求；
- 失败、重试和补偿方式；
- 兼容范围；
- 双方责任。

契约不是单纯的技术接口字段列表。

### 4.4 场景 Scenario

场景表达一次完整、可观察的业务结果。

一个场景至少具有：

- 一个明确触发；
- 一个主要业务目标；
- 一条主路径及必要的异常路径；
- 一个可观察的最终结果；
- 一个横向结果负责人。

只有当一条分支拥有独立目标、生命周期或负责人时，才考虑拆成独立场景。

### 4.5 不变量 Invariant

不变量表达业务变化过程中必须始终成立的业务承诺。

领域不变量由单个领域负责，例如：

```text
一次性权益不能重复消费
订单支付成功后不能再次支付
游戏状态只能沿合法路径迁移
```

端到端不变量属于横向业务场景，例如：

```text
一次成功支付最终对应一次有效权益
任何一次业务结果都能追溯到合法来源
```

端到端不变量需要拆分为各领域的局部责任和契约责任。

## 5. 核心关系

五类业务对象通过稳定引用建立关系：

```text
领域拥有能力
场景使用能力
能力通过契约协作
不变量约束领域、契约或场景
验证证据证明不变量或场景结果
业务变化修改对象及其关系
```

例如：

```text
SCN-PAID-REVIVE 使用 CAP-PAYMENT
SCN-PAID-REVIVE 使用 CAP-ENTITLEMENT
CAP-ENTITLEMENT 属于 DOM-COMMERCE
CTR-REVIVE-ENTITLEMENT 连接商业化与可玩内容
INV-ENTITLEMENT-ONCE 约束 CAP-ENTITLEMENT
```

稳定标识的目的不是将业务变成编码体系，而是让事实可以被准确引用，避免在不同文档中复制定义。

## 6. 业务变化的最小表达

一次业务变化最少需要回答四类问题：

```text
业务变化
= 变化意图
+ 模型差异
+ 生效条件
+ 正确性承诺
```

这是四类必须表达的业务语义，不是固定字段模板。

### 6.1 变化意图：为什么变化

需要能够理解：

- 当前存在什么问题；
- 为什么值得改变当前业务；
- 预期产生什么业务结果；
- 哪些内容不是本次目标。

### 6.2 模型差异：改变了什么

变化可以作用于领域、能力、契约、场景、不变量及其关系。

从形成新状态的角度，最小状态操作只有：

```text
新建 / 修改 / 失效
```

“复用、扩展、替换、绕过”等描述变化性质，不是同一层的状态操作。

例如：

```text
修改 CAP-ENTITLEMENT
变化性质：扩展
业务含义：支持一次性复活权益

新建 CTR-REVIVE-ENTITLEMENT
变化性质：正式新增
```

### 6.3 生效条件：何时、对谁有效

至少需要表达：

- 生命周期状态；
- 生效时间或生效事件；
- 生效范围；
- 失效时间或退出条件。

生命周期使用最小语义：

```text
提议 → 已确认 → 已生效 → 已失效
```

生效范围可以包括用户分组、地区、租户、客户端版本、服务版本、功能开关或业务配置。

因此，有效业务模型不是简单的 `M(时间)`，而是：

```text
M(时间, 生效上下文)
```

### 6.4 正确性承诺：如何证明正确

需要表达：

- 新增或修改了哪些不变量；
- 哪些历史不变量可能受到影响；
- 横向场景必须得到什么结果；
- 使用哪些证据证明正确；
- 当前是否存在验证缺口。

证据可以是自动化测试、人工验收、业务对账、指标、监控、灰度观察、演练记录或决策确认。

## 7. 从变化形成新的业务状态

业务模型在某个时间和上下文中的有效状态，可以理解为：

```text
有效业务模型
= 基础业务模型
+ 所有已经生效且适用于当前上下文的业务变化
```

基本过程为：

```text
提出业务变化
→ 说明变化意图
→ 描述目标业务状态
→ 建立对象和关系差异
→ 明确生效条件
→ 明确正确性承诺
→ 实施与验证
→ 在适用范围内生效
→ 形成新的当前状态
→ 保留变化记录
```

未生效变化可以用于展示未来状态，但不能混入当前有效模型。

灰度期间新旧状态可以同时存在，由时间、版本、用户范围和功能开关等上下文决定当前适用的模型。

对象下线后应标记失效并保留历史引用，不直接删除历史业务事实。

## 8. 四种派生视图

### 8.1 横向场景视图

从场景出发，展开：

```text
场景
→ 使用的领域能力
→ 能力之间的契约
→ 端到端不变量
→ 最终业务结果
```

它回答当前业务如何跨领域闭环。

### 8.2 纵向领域视图

从领域出发，展开：

```text
领域
→ 拥有的能力
→ 领域不变量
→ 对外契约
→ 使用这些能力的业务场景
```

它回答领域沉淀了什么，以及能力被如何复用。

### 8.3 版本变化视图

版本是多个业务变化的交付集合：

```text
版本
→ 包含的业务变化
→ 直接修改的对象和关系
→ 生效条件
→ 兼容、灰度、降级和回滚方式
```

版本视图只描述差异，不复制完整业务现状。

### 8.4 验证视图

从变化对象反向推导：

```text
变化影响的领域不变量 → 纵向回归
变化影响的契约       → 集成验证
变化影响的业务场景   → 横向验收
变化的生效方式       → 灰度、兼容和回滚验证
```

工具或人工可以生成候选影响，但最终验证判断仍需要结合业务语义确认。

## 9. 正确性的五个层次

| 层次 | 核心问题 | 主要证据 |
| --- | --- | --- |
| 领域正确性 | 单个领域内部是否自洽 | 规则校验、状态机测试、领域回归 |
| 契约正确性 | 领域之间是否正确理解彼此 | 契约测试、幂等和异常协议验证 |
| 场景正确性 | 业务流程是否完整闭环 | 端到端验收、异常场景、业务对账 |
| 演进正确性 | 新旧状态共存和切换是否安全 | 兼容测试、迁移验证、回滚演练 |
| 观测正确性 | 异常能否被发现和恢复 | 标识、指标、告警、对账和补偿 |

版本可发布的最低逻辑为：

```text
受影响领域回归通过
+ 变更契约验证通过
+ 关键业务场景通过
+ 生效、降级和回滚路径通过
+ 必要观测和补偿能力就绪
```

兼容范围需要显式规定，不要求永久支持所有历史版本的任意组合。

## 10. 单一事实与开放表达

### 10.1 双层表达

为了同时满足可维护性和业务表达自由，采用双层模型。

结构化内核只负责：

```text
稳定身份
对象类型
对象关系
变化动作
生命周期状态
生效条件
引用关系
```

开放语义正文负责：

- 背景和判断；
- 规则解释；
- 正常、异常和降级场景；
- 状态变化；
- 取舍与被放弃的方案；
- 示例、反例、图表和附件；
- 暂时无法结构化的新业务内容。

> 结构化内核负责连接，开放语义正文负责表达。

### 10.2 避免重复维护

1. 事实只在其所有者对象中定义；
2. 其他对象使用稳定标识引用，不复制正文；
3. 场景表达能力组合，不重新定义能力；
4. 版本记录变化，不复制当前模型；
5. 验证证据关联业务承诺，不孤立维护测试清单；
6. 派生视图不直接编辑，错误应回到源事实修正。

### 10.3 开放性约束

- 先表达业务事实，再考虑如何结构化；
- 不建立封闭字段全集；
- 新内容无法被当前结构理解时，允许先完整保存；
- 机器推导提供候选影响，不替代人的业务判断；
- Schema 服务于业务模型，业务模型不服务于 Schema。

## 11. 责任结构

### 11.1 场景负责人

对端到端业务目标和横向业务结果负责。

### 11.2 领域负责人

对领域内部概念、能力、规则、状态和历史逻辑负责。

### 11.3 契约双方

共同对领域交界处的业务语义、异常处理和兼容性负责。

跨领域不变量首先属于场景，由场景负责人承担整体责任；各领域负责人承担被拆分后的局部责任。

## 12. 在产品研发过程中的使用

### 12.1 需求提出

- 说明为什么变化；
- 描述变化前后可观察的业务差异；
- 明确业务目标和非目标。

### 12.2 方案设计

- 建立或修改横向业务场景；
- 标记涉及的领域能力；
- 明确契约和不变量变化；
- 描述数据、兼容和生效条件。

### 12.3 开发与验证

- 根据领域影响形成纵向回归；
- 根据契约变化形成集成验证；
- 根据场景变化形成横向验收；
- 根据生效条件形成灰度、兼容和回滚验证。

### 12.4 发布完成

- 在适用范围内将变化标记为生效；
- 更新当前业务模型的有效状态；
- 保留业务变化和验证证据；
- 记录临时绕过、验证缺口和退出条件。

具体业务变更的表达方式可参考配套文档：[业务变更 Spec 定义与模板](../02-业务变更/业务变更Spec定义与模板.md)。该模板是表达辅助，不是强制 Schema。

## 13. 简化示例：玩家失败后付费复活

### 13.1 变化意图

部分高意愿玩家在挑战失败后仍希望继续。目标是验证付费复活对关卡完成率和付费转化的影响。

### 13.2 模型差异

- 扩展可玩内容领域的挑战状态恢复能力；
- 扩展商业化领域的一次性权益能力；
- 新增商业化到可玩内容的复活权益契约；
- 新增玩家失败后付费复活场景；
- 新增“一次成功支付最终对应一次有效复活”的端到端不变量。

### 13.3 生效条件

新场景仅对支持该能力的客户端和实验用户生效，由功能开关控制。灰度期间旧场景继续有效。

### 13.4 正确性承诺

- 一次支付最多产生一份复活权益；
- 一份权益最多完成一次复活；
- 支付成功但后续失败时可以被发现和补偿；
- 旧客户端继续运行旧场景；
- 关闭入口后不再产生新订单，但已有订单继续正确处理。

由此可以从同一份事实得到：

- 横向：支付到游戏恢复的完整路径；
- 纵向：商业化和可玩内容分别发生的能力变化；
- 版本：本次交付包含的对象变化与灰度范围；
- 验证：领域回归、契约验证、端到端验收和灰度回滚检查。

## 14. V1 的方法边界

### 14.1 V1 负责

- 表达当前业务状态；
- 表达跨领域业务场景；
- 表达领域能力及其沉淀；
- 表达一次业务变化及其生效条件；
- 表达必须保持的业务正确性；
- 从同一份事实形成横向、纵向、版本和验证视图。

### 14.2 V1 暂不负责

- 给出所有领域的标准划分答案；
- 自动完成影响分析和测试生成；
- 衡量领域成熟度；
- 设计完整的领域债务治理机制；
- 规定领域边界重构方法；
- 决定使用何种工具或平台；
- 为所有特殊业务建立预设字段。

## 15. V1 的完成与验证

理论结构到此停止扩展。V1 接下来只通过真实业务案例验证，不再增加假设性概念。

验证过程为：

```text
描述变化前的当前业务状态
→ 使用四类语义表达一次真实变化
→ 形成特定范围下的新业务状态
→ 从同一份事实得到四种视图
→ 检查是否出现重复维护或无法表达的内容
```

V1 的实践完成标准是能够稳定回答：

1. 当前业务如何运行？
2. 本次业务为什么变化？
3. 具体改变了哪些能力、规则和连接？
4. 变化在何时、什么范围内有效？
5. 如何证明变化后的业务仍然正确？

如果真实案例能够完整走通，V1 即完成实践闭环。后续只有在同类问题反复出现、现有模型确实无法表达时，才进入下一版本。

## 16. 最终总结

横向、纵向和时间不是三套独立管理体系，而是同一个演进中业务模型的三个投影：

```text
横向场景保证业务价值闭环
纵向领域保证能力长期积累
业务变化推动模型随时间演进
不变量和证据保证变化正确
```

整套方法最终可以压缩为：

```text
当前业务模型
+ 一次开放表达的业务变化
+ 明确的生效条件
→ 特定上下文下的新业务状态
→ 横向、纵向、版本和验证视图
```

方法论追求的不是把全部业务知识完全结构化，而是：

> 在不损失业务表达自由的前提下，为业务事实建立足以连接、追踪、演进和验证的最小结构。
