# 横纵时间方法论的范围与收敛

> 状态：收敛稿  
> 目的：明确方法论应该解决什么、不应该继续扩展什么，避免问题无限发散。

## 1. 当前需要解决的核心问题

在前面的探索中，形成了领域、能力、场景、契约、版本、业务变更、不变量、验收和工具表达等一系列问题。

如果继续逐项深入，方法论很容易扩展成一套试图解决所有领域设计、组织治理、测试自动化和工具建设问题的庞大体系，最终难以落地。

因此，当前首先需要确定这套方法论的边界。

核心判断是：

> 方法论负责提供统一语法，各领域负责填充具体内容。

方法论不需要给出每个领域的标准答案，但必须保证不同领域产生的内容可以使用同一套方式表达、连接、演进和验证。

## 2. 对待探索问题的重新归类

此前提出了八个后续问题：

1. 如何判断什么应该成为领域，什么只是一项能力？
2. 业务场景应该拆到什么颗粒度？
3. 如何从业务变更自动推导影响范围和测试范围？
4. 如何衡量领域是否形成能力沉淀，而不是功能堆积？
5. 如何处理跨领域不变量的最终责任归属？
6. 如何把临时方案和领域债务纳入治理？
7. 什么情况下应该调整领域边界？
8. 如何用工具生成不同视图，同时避免重复维护？

这些问题并不完全属于同一层次。

| 问题 | 是否属于核心方法论 | 当前处理方式 |
| --- | --- | --- |
| 领域与能力如何区分 | 是，但只定义判断原则 | 各领域负责具体划分 |
| 场景颗粒度如何确定 | 是，需要统一最低约定 | 不研究所有特殊情况 |
| 自动推导影响和测试范围 | 否 | 先支持人工推导，自动化后置 |
| 衡量能力沉淀程度 | 否 | 作为成熟度评估后置 |
| 跨领域不变量的责任 | 是 | 必须明确最低责任结构 |
| 临时方案和领域债务 | 部分属于 | 方法只要求显式记录，完整治理后置 |
| 何时调整领域边界 | 非核心 | 作为领域演进方法后置 |
| 多视图与重复维护 | 原则属于，工具不属于 | 确立单一事实、多种视图原则 |

当前不再将这八项分别扩展成独立研究方向，只确定维持方法成立所必需的最低规则。

## 3. 领域与能力的最低定义

方法论不需要提供一套能够自动产生完美领域边界的算法，但必须统一“领域”和“能力”的语义。

### 3.1 领域

领域是业务责任和业务事实的所有权边界。

一个领域通常：

- 拥有一组相对稳定的业务概念；
- 对一组业务数据和状态负责；
- 维护自己的规则和不变量；
- 能够独立解释自身为什么这样运行；
- 对外提供可以被业务场景使用的能力。

### 3.2 能力

能力是领域基于自己拥有的事实、状态和规则，对外完成的一种业务行为或判断。

两者的关系可以表达为：

```text
领域拥有事实、规则和状态
能力基于这些事实、规则和状态完成某种行为
```

例如：

```text
商业化：领域
购买资格判断：能力
订单创建：能力
权益发放：能力
```

方法论只需要保证这层区别成立。具体哪些概念构成一个领域、哪些行为构成领域能力，由实际业务进行判断。

## 4. 业务场景颗粒度的最低定义

业务场景以“一次完整业务结果”为基本边界。

一个场景至少具有：

- 一个明确触发；
- 一个主要业务目标；
- 一个可观察的业务结果；
- 一个横向结果负责人。

例如，“玩家失败后付费复活”可以作为一个场景：

- 触发：玩家挑战失败；
- 目标：允许有意愿的玩家通过付费继续挑战；
- 结果：玩家获得合法复活机会并恢复到正确状态；
- 负责人：对该端到端业务结果负责的人。

支付失败、权益发放超时等属于该场景的异常分支，不需要分别建立场景。

只有当一条分支拥有独立的业务目标、生命周期或负责人时，才考虑拆成新的场景。

这项最低约定足以控制场景颗粒度，不需要继续研究所有可能的特殊情况。

## 5. 跨领域不变量的责任规则

跨领域不变量不应被强行归入某一个领域。它首先属于横向业务场景。

最低责任结构为：

```text
场景负责人
  对端到端业务结果负责

领域负责人
  对本领域承担的局部规则负责

契约双方
  对领域交界处的业务语义和兼容性共同负责
```

例如：

```text
端到端不变量：
一次成功支付最终对应一次有效复活

商业化：
保证支付与权益发放唯一对应

可玩内容：
保证权益幂等消费和游戏状态正确恢复

领域契约：
保证订单、权益和挑战使用统一业务标识

场景负责人：
保证局部责任组合后，端到端业务结果成立
```

这项规则解决的是横向业务结果的责任结构，不继续扩展为完整的跨团队治理体系。

## 6. 临时方案的最低管理要求

当前方法论不建立完整的领域债务治理体系。

但当一次业务变化被标记为“绕过”或“临时方案”时，必须显式记录：

```text
为什么采用临时方案
绕过了什么正式能力或规则
产生什么业务和系统风险
影响哪些业务场景
由谁负责后续处理
退出条件是什么
```

这项要求的目标是让临时变化不再依赖个人记忆。

至于临时方案的优先级、考核、偿还计划和组织机制，属于后续治理问题，不纳入当前核心方法论。

## 7. 单一事实与多种视图

“使用什么工具自动生成视图”不是方法论本身，但以下原则属于方法论核心：

> 领域、能力、契约、场景和版本不能通过复制内容分别维护，必须来自同一份业务事实。

同一份业务事实可以产生：

- 横向业务场景视图；
- 纵向领域能力视图；
- 时间版本和变化视图；
- 影响分析与验证视图。

至于底层使用：

- Markdown；
- YAML；
- 数据库；
- 知识图谱；
- 自研业务建模平台；

都不影响方法论成立。

因此，当前确定的是“单一事实、多种投影”的表达原则，不继续展开工具选型和平台设计。

## 8. 收敛后的最小模型

### 8.1 纵向

```text
领域
  → 拥有业务事实、状态和规则
  → 对外提供业务能力
  → 通过领域不变量保证内部正确
```

### 8.2 横向

```text
业务场景
  → 调用多个领域能力
  → 通过领域契约连接
  → 通过端到端不变量保证整体正确
```

### 8.3 时间

```text
当前业务模型
  → 业务变更集
  → 新的当前业务模型
```

每项业务变更只需要标记其对已有模型的影响性质：

```text
复用 / 扩展 / 修改 / 新建 / 绕过 / 废弃
```

### 8.4 验证

```text
领域不变量 → 纵向回归
领域契约   → 集成验证
业务场景   → 横向验收
版本变化   → 兼容、灰度和回滚验证
```

## 9. 方法论的明确范围

### 9.1 当前负责解决

这套方法论负责：

- 用横向场景表达业务价值；
- 用纵向领域表达能力沉淀；
- 用契约表达横纵交界；
- 用不变量表达必须保持的业务正确性；
- 用变更集表达时间演进；
- 用验收证据验证变化是否正确；
- 保证一份业务事实可以形成多种视图。

### 9.2 当前不负责解决

这套方法论暂时不负责：

- 给出所有领域的标准划分答案；
- 自动完成影响分析和测试生成；
- 衡量领域成熟度；
- 设计完整的领域债务治理机制；
- 规定领域边界重构的方法；
- 决定具体使用什么工具和平台。

这些问题可以在方法论实际应用后，根据真实阻力继续补充，而不是在第一阶段提前展开。

## 10. 最终收敛定义

这套方法论不是一套“如何设计所有业务领域”的方法，而是：

> 一套使用统一结构描述业务当前状态、跨领域运行方式以及持续变化的方法。

它需要稳定回答四个问题：

1. 当前业务通过哪些领域和能力运行？
2. 一个完整业务场景如何横向闭环？
3. 本次变化修改了哪些能力、规则和连接？
4. 如何证明变化后的业务仍然正确？

只要能够持续回答这四个问题，横向、纵向和时间三个维度就已经形成完整闭环。其他问题应被视为具体领域实践、组织治理或工具建设，而不再无限扩展当前方法论的边界。
