# 业务演进与 AI Native 方法论

这里集中保存两类相互关联、但成熟度不同的思考：

1. 如何用横向场景、纵向领域和时间变化统一表达业务；
2. 如何让组织、人和智能体基于这些业务事实协同运行，并进一步形成 AI Native 生产系统。

目录按文档角色组织，而不是按创建时间平铺。V1 和配套规范放在前面，仍在发散的问题进入讨论稿，已经被后续文档吸收的内容保留为演进记录。

## 建议阅读顺序

1. [横纵时间业务演进方法论 V1](<./01-核心框架/横纵时间业务演进方法论 V1.md>)：当前总纲和统一语言。
2. [最小业务变化表达与开放性原则](./01-核心框架/最小业务变化表达与开放性原则.md)：一次变化最少需要表达什么。
3. [统一业务模型的变化与视图](./02-业务变更/统一业务模型的变化与视图.md)：变化如何写入统一事实，不同视图如何从中投影。
4. [业务变更 Spec 定义与模板](./02-业务变更/业务变更Spec定义与模板.md)：把方法论带入具体需求和交付过程。
5. [组织、人和智能体的能力沉淀](./03-组织与智能体/组织、人和智能体的能力沉淀.md)：从业务对象扩展到行动主体和可执行记忆。
6. [从横纵方法论到演化型组织](./03-组织与智能体/从横纵方法论到演化型组织.md)：继续讨论价值、环境、权力、证据和学习。

## 目录说明

```text
methodology/
├── README.md
├── 01-核心框架/       # 当前应优先阅读和引用的总纲、核心原则
├── 02-业务变更/       # 统一模型、变化机制和 Spec 表达
├── 03-组织与智能体/   # 组织、人、智能体及演化型组织的扩展
├── 04-讨论稿/         # 尚未收敛的探索，以及外部讨论快照
└── 05-演进记录/       # 被 V1 吸收或取代、但仍有推演价值的历史稿
```

### 01-核心框架

- `横纵时间业务演进方法论 V1.md`：当前方法论主入口。
- `最小业务变化表达与开放性原则.md`：V1 核心结构候选，强调语义约束而非固定 Schema。

### 02-业务变更

- `统一业务模型的变化与视图.md`：统一业务事实的写入、演进与读取。
- `业务变更Spec定义与模板.md`：业务变化在需求推进和交付中的具体表达。

### 03-组织与智能体

- `组织、人和智能体的能力沉淀.md`：组织可执行记忆及行动主体模型。
- `从横纵方法论到演化型组织.md`：方法论边界之外的组织演化议题。

### 04-讨论稿

- `业务模型之外的组织实现与环境扩展.md`：Organization、Realization、Ecosystem 等关联模型的边界草案。
- `AI Native生产系统讨论稿.md`：从状态、上下文、Spec、Agent、Evidence、Evaluation 和 Memory 等角度讨论 AI Native 生产系统。

AI Native 文档是从 `ai-production-os/owner/生产系统.md` 收录的讨论快照，用于保存方法论之间的连接与推演过程。AI Native 系统的当前实现、协议和产品定义仍以 `ai-production-os` 仓库为准，避免在这里形成第二份事实源。

### 05-演进记录

这里保留 V1 形成前的初稿、闭环探索、范围收敛和路线讨论。它们用于理解判断是如何形成的，不作为当前结论的首选引用。

## 维护约定

- 新结论先判断它属于核心框架、配套模型还是开放讨论，不再直接堆到根目录。
- 讨论稿收敛后移动到对应主题目录，并在这里更新阅读顺序。
- 新版本取代旧版本时，旧稿进入 `05-演进记录/`，不直接删除。
- 跨仓库收录的材料必须标明来源和事实源，避免快照与运行中的系统定义混淆。
