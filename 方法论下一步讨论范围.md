# 方法论下一步讨论范围

> 状态：路线稿  
> 目的：明确哪些内容已经收敛、哪些问题仍需讨论，以及方法论第一版应在何处停止扩展。

## 1. 当前判断

横向、纵向和时间的概念层已经基本收敛。

当前缺少的不是更多概念，而是把已有概念进一步压缩成一套可以实际运行的协议。

如果继续扩展新的理论问题，方法论很容易从“统一表达业务及其变化”扩大为一套覆盖领域设计、组织治理、测试自动化和工具平台的庞大体系，最终难以落地。

因此，后续不再广泛增加新的研究方向，而只讨论使方法论能够运行所必需的问题。

## 2. 已经基本确定的内容

以下内容已经形成相对稳定的判断，不需要继续反复讨论：

- 为什么需要横向、纵向和时间三个视角；
- 横向用于表达跨领域业务闭环；
- 纵向用于表达领域内部的能力沉淀；
- 时间用于表达业务模型从一个状态到另一个状态的演进；
- 领域、能力、场景、契约、变化和不变量的基本含义；
- 横向正确、纵向正确和演进正确的区别；
- 场景负责人、领域负责人和契约双方的最低责任关系；
- 方法论负责什么、不负责什么；
- 单一事实、多种视图的基本原则。

这些内容可以作为后续讨论的稳定前提。

## 3. 仍需讨论的问题一：单一事实由什么构成

目前已经确定“单一事实、多种视图”的原则，但尚未最终确定：

- 哪些内容是需要长期维护的源数据；
- 哪些内容只是从源数据生成的视图；
- 哪些概念应该成为独立对象；
- 哪些概念只应该作为对象的属性存在。

### 3.1 候选源数据

当前可以将源数据初步收敛为六类：

```text
领域 Domain
能力 Capability
契约 Contract
场景 Scenario
不变量 Invariant
变更 Change
```

### 3.2 固定关系

六类源数据之间的基本关系是：

```text
领域拥有能力
能力通过契约协作
场景引用多个能力和契约
不变量约束领域、契约或场景
变更修改上述业务对象
```

### 3.3 生成视图

基于上述源数据，可以生成：

```text
纵向领域能力视图
横向业务场景视图
时间版本与变化视图
影响分析视图
回归与验收视图
```

下一步需要对六类源数据做一次最终检查：

1. 是否足够表达横向、纵向和时间；
2. 是否存在概念重复；
3. 不变量是否应作为独立对象；
4. 领域是否只是能力的容器，还是拥有独立业务语义；
5. 版本是否需要成为独立对象，还是仅作为多个变更的发布集合；
6. 业务目标应作为独立对象，还是场景和变更的属性。

这部分定义的是方法论的统一语法，因此必须在第一版中确定。

## 4. 仍需讨论的问题二：业务变化的生命周期

时间维度目前已经有概念模型，但还缺少明确的运行规则。

需要定义一次业务变化如何进入系统、如何被实施，以及如何最终转化为新的当前业务状态。

### 4.1 基本过程

```text
需求提出
→ 建立业务变更
→ 描述目标业务场景
→ 标记领域能力和契约变化
→ 定义不变量和验证方式
→ 实施与发布
→ 合并到当前有效模型
→ 保留历史变化记录
```

### 4.2 候选生命周期

```text
提议中
→ 已确认
→ 实施中
→ 已发布
→ 已废弃
```

生命周期不只是项目状态，还决定某项业务变化是否已经成为当前有效业务模型的一部分。

### 4.3 需要确定的规则

- 未发布内容是否进入当前业务模型；
- 已确认但尚未实施的内容如何展示；
- 一个业务变更跨多个版本时如何表示；
- 一个版本包含多个业务变更时如何组织；
- 发布后，场景、能力、契约和不变量如何形成新的当前状态；
- 旧场景和旧能力下线后，是删除还是标记失效；
- 灰度期间新旧业务状态如何同时表达；
- 发布失败或发生回滚时，当前模型如何恢复；
- 历史变化记录如何与当前有效模型保持关联。

这部分定义的是时间维度真正的落地方式。

## 5. 用真实案例代替继续抽象扩展

完成上述两个问题的最低定义后，不应该继续研究新的理论分支，而应该选择一个真实业务变化，完整走一遍方法论。

建议验证过程为：

```text
1. 描述变化前的当前业务状态
2. 建立一个业务变更
3. 表达变化后的目标业务场景
4. 标记涉及的领域和能力变化
5. 记录跨领域契约和不变量变化
6. 形成版本影响与验收范围
7. 模拟灰度、发布和回滚
8. 模拟发布后如何更新当前有效模型
```

实例验证的重点不是把案例写得极其完整，而是检查方法论能否稳定回答：

1. 当前业务如何运行？
2. 本次业务为什么变化？
3. 具体改变了哪些能力、规则和连接？
4. 如何证明变化后的业务仍然正确？

如果真实案例无法被当前对象表达，再根据实际阻力补充模型；不再通过假设所有可能问题提前扩展方法论。

## 6. 当前暂不讨论的内容

以下内容不纳入方法论第一版的继续讨论范围：

- 自动完成影响分析；
- 自动生成测试；
- 领域成熟度评分；
- 完整的领域债务治理；
- 领域边界重构方法；
- 完整平台和可视化系统；
- 所有特殊业务场景的建模规则；
- 针对所有组织形态的责任治理方案。

这些内容只有在真实案例验证中成为明确阻力时，才重新进入讨论。

## 7. 方法论第一版的停止条件

当方法论能够稳定完成以下事情时，第一版应该停止扩展：

- 使用同一份源数据表达当前业务状态；
- 从横向看清一个业务场景如何跨领域闭环；
- 从纵向看清每个领域拥有并积累了哪些能力；
- 使用业务变更表达新旧业务状态之间的差异；
- 根据不变量、契约和场景形成基本验证范围；
- 发布后能够形成新的当前业务状态，同时保留历史变化；
- 不需要通过复制内容维护横向、纵向和时间视图。

第一版不以覆盖所有业务问题为完成标准，而以形成一个最小、完整、能够实际运行的闭环为完成标准。

## 8. 建议的下一步顺序

```text
第一步：确定六类源数据及其关系
第二步：确定业务变化的生命周期
第三步：用一个真实案例完成全流程验证
第四步：根据实例结果进行一次必要修正
第五步：将探索文档合并为一份方法论 V1
```

因此，下一阶段可以收敛为一个总问题：

> 一次业务变化，如何基于统一的源数据，从当前业务状态进入系统，并最终形成新的当前业务状态？

这个问题解决后，横向、纵向和时间三个维度就形成了一个可以实际运行的完整闭环。
