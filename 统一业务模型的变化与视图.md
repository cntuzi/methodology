# 统一业务模型的变化与视图

> 状态：核心模型探索稿  
> 目的：统一解释业务变化如何形成新的当前状态，以及如何由同一份业务事实生成横向、纵向、版本和验证视图。

## 1. 两个问题实质上是同一个问题

此前存在两个待讨论问题：

1. 如何用工具生成横向、纵向、版本和验证视图，同时避免重复维护？
2. 一次业务变化如何基于统一的源数据，从当前业务状态进入系统，并最终形成新的当前业务状态？

这两个问题共享同一个核心，但分别对应统一业务模型的两个方向：

- 业务变化如何形成新的当前状态，是模型的写入与演进问题；
- 如何生成不同视图，是模型的读取与投影问题。

因此，它们不应该作为两个相互独立的方法论问题分别解决。

真正的核心问题是：

> 如何定义一个能够持续演进的统一业务模型，使业务变化可以修改它，并可以从中推导不同视角？

关键不在工具，而在以下四件事：

1. 什么是唯一业务事实；
2. 一次业务变化如何描述；
3. 业务变化在什么条件下生效；
4. 不同视图如何从业务事实和变化中推导。

如果这四件事没有定义清楚，使用知识图谱、数据库或自研平台，也只会得到一个更加复杂的文档仓库。

反过来，如果这四件事已经明确，第一版即使使用 Markdown 和对象引用，也能够成立。

## 2. 统一模型

假设某个时间的当前业务模型为：

```text
M(t)
```

它包含当前有效的：

```text
领域
能力
契约
场景
不变量
```

现在发生一次业务变化：

```text
Δ
```

这次变化需要记录：

```text
新增什么
修改什么
复用什么
绕过什么
废弃什么
为什么变化
何时生效
对谁生效
如何验证
```

变化生效后：

```text
M(t+1) = Apply(M(t), Δ)
```

各种视图只是对业务模型和变化的不同投影：

```text
横向视图 = ProjectByScenario(M)
纵向视图 = ProjectByDomain(M)
版本视图 = ProjectByChange(M, Δ)
验证视图 = ProjectByImpact(M, Δ)
```

这套关系同时解释了业务状态演进与多视图生成。

## 3. 四种视图不是四份数据

横向、纵向、版本和验证不应该分别维护完整内容。它们是同一组业务事实的四种读取方式。

### 3.1 横向视图

横向视图从某个业务场景开始，查询它引用的能力、契约和端到端不变量：

```text
业务场景
→ 涉及的领域能力
→ 能力所属领域
→ 能力之间的契约
→ 端到端不变量
→ 最终业务结果
```

它回答：

- 一个业务目标如何跨领域形成闭环；
- 场景经过哪些能力；
- 领域在什么位置发生交接；
- 整体业务结果必须满足什么约束。

### 3.2 纵向视图

纵向视图从某个领域开始，查询它拥有的能力、规则、契约，以及这些能力被哪些场景引用：

```text
领域
→ 拥有的能力
→ 领域规则和不变量
→ 对外契约
→ 被哪些业务场景使用
```

它回答：

- 一个领域当前拥有什么；
- 领域内部如何保持自洽；
- 领域能力如何被不同业务场景复用；
- 一次变化对领域的长期积累产生什么影响。

### 3.3 版本视图

版本视图从一组业务变化开始，查询它修改了什么：

```text
版本
→ 包含的业务变化
→ 新增、修改、复用、绕过和废弃的对象
→ 生效时间与生效范围
→ 兼容、灰度、降级和回滚方式
```

版本视图不复制完整业务现状，只描述这次交付与已有业务模型之间的差异。

### 3.4 验证视图

验证视图根据业务变化影响的对象反向推导：

```text
变化影响的领域不变量
→ 纵向领域回归

变化影响的契约
→ 跨领域集成验证

变化影响的业务场景
→ 横向业务验收

变化的生效方式
→ 灰度、兼容、迁移和回滚验证
```

验证视图回答的不是“测试团队准备了哪些用例”，而是“根据业务模型的变化，哪些正确性承诺必须重新得到证明”。

## 4. 避免重复维护的规则

### 4.1 事实只定义在所有者对象中

例如，“购买资格判断属于商业化领域”只在能力对象中定义：

```text
能力 ID：CAP-PURCHASE-ELIGIBILITY
所属领域：DOM-COMMERCE
```

业务场景中只引用：

```text
使用能力：CAP-PURCHASE-ELIGIBILITY
```

场景不重新复制能力的完整说明。

### 4.2 场景只表达组合关系

场景负责描述多个领域能力如何协作形成业务闭环，不负责重新定义每个能力内部如何运行。

如果某项能力的业务规则发生变化，应修改能力或不变量对象，而不是在所有引用该能力的场景中分别修改。

### 4.3 版本只记录变化

版本记录：

```text
新增：SCN-PAID-REVIVE
扩展：CAP-ENTITLEMENT
新增：CTR-REVIVE-ENTITLEMENT
```

版本不重新复制商业化领域、可玩内容领域和完整业务场景。

### 4.4 验证证据关联业务承诺

测试、对账和监控证据通过引用关联不变量、契约、场景和变化：

```text
REG-COM-012
→ 验证 INV-ENTITLEMENT-ONCE

E2E-REVIVE-003
→ 验证 SCN-PAID-REVIVE

COMPAT-008
→ 验证 CHG-PAID-REVIVE 的灰度兼容
```

验证视图通过关联关系生成，不重新维护一份彼此孤立的测试范围说明。

### 4.5 派生视图不直接编辑

如果横向视图中的能力名称错误，应修改能力对象，而不是直接修改横向图。

如果版本视图缺少某项影响，应补充业务变化与业务对象之间的关系，而不是直接修补版本报告。

如果验证视图缺少某项回归，应补充不变量、场景和验证证据的关系，而不是只在测试清单中临时增加一行。

因此：

> 源数据可以编辑，派生视图只能通过修改源数据和关系发生变化。

这是避免重复维护最关键的约束。

## 5. 当前业务状态不一定唯一

简单的 `M(t)` 只能表达某个时间点的单一业务状态，但生产环境经常同时存在多种有效状态。

例如灰度期间可能出现：

- 10% 用户运行新场景，90% 用户运行旧场景；
- 新客户端支持新能力，旧客户端仍然运行旧逻辑；
- 某些国家和地区已经生效，其他地区尚未生效；
- 某个租户启用了功能，其他租户未启用；
- 同一版本受到不同功能开关控制。

所以业务模型更准确地应表示为：

```text
M(时间, 生效范围)
```

一次业务变化至少需要携带：

```text
生命周期状态
生效时间
生效范围
用户分组
地区或租户
客户端版本
功能开关
```

并不是所有变化都必须使用所有条件，但模型必须允许表达这些条件。

因此，“当前业务如何运行”实际上是在询问：

> 在某个时间、某个地区、某个客户端版本和某个灰度范围内，哪些业务变化有效？

工具未来可以提高查询和渲染效率，但“业务变化具有生效条件”属于方法论本身。

## 6. 统一后的核心结构

两个问题可以合并为以下模型：

```text
当前业务模型 M
    +
业务变化 Δ
    +
生效条件 E
    ↓
特定时间和范围下的有效业务模型
    ↓
横向视图 / 纵向视图 / 版本视图 / 验证视图
```

其中：

- 源数据：业务对象及其关系；
- 变化：对业务对象及其关系的修改；
- 生效条件：变化何时、在什么范围内有效；
- 有效模型：特定时间和范围下实际运行的业务状态；
- 视图：根据不同问题对有效模型进行投影。

## 7. 工具的正确位置

工具不负责决定方法论，只负责执行已经确定的语义。

工具可以负责：

- 保存业务对象及其关系；
- 为对象分配和校验稳定标识；
- 记录和应用业务变化；
- 根据时间和范围选择有效变化；
- 查询对象之间的影响关系；
- 渲染不同视图；
- 检查引用缺失、重复定义和失效关系。

工具不能替代以下方法论决策：

- 什么属于领域事实；
- 什么构成一个业务能力；
- 什么是完整业务场景；
- 哪些变化会改变业务语义；
- 什么不变量必须保持；
- 一项业务变化在什么范围内有效。

所以工具建设应该发生在业务模型、变化语义和投影规则确定之后。

## 8. 下一步问题的进一步收敛

后续不再分别讨论：

- 如何生成横向、纵向、版本和验证视图；
- 一次变化如何形成新的业务状态。

它们可以合并为一个问题：

> 一次业务变化应该使用什么最小结构表达，才能既形成新的有效业务状态，又自然推导出横向、纵向、版本和验证视图？

回答这个问题需要确定三项最小内容：

1. 当前业务模型包含哪些基本对象和关系；
2. 业务变化如何修改这些对象和关系，并在什么条件下生效；
3. 各种视图分别按照什么规则进行投影。

当这三项内容能够通过一个真实业务案例完整运行时，横向、纵向和时间方法论就达到了第一版的核心停止条件。
