# 业务变更与横纵闭环

> 状态：探索稿  
> 与《横纵业务演进方法论》的关系：前者定义基础表达对象和视图，本文进一步探索如何让这些对象进入需求分析、方案设计、交付验收和持续演进过程。

## 1. 从“知识表达”走向“业务推进”

领域、能力、场景、契约和版本可以帮助描述业务，但仅有这些对象，还不足以保证方法论真正参与日常工作。

更关键的问题是：

> 如何让横纵模型直接参与需求分析、方案决策、研发推进和验收，而不是成为一套事后整理的文档？

解决这个问题的突破口，是把“变化”提升为一等对象。

业务系统中存在两类不同的信息：

- 稳定信息：当前业务如何运行；
- 变化信息：为什么要改变、改变了什么，以及如何证明变化正确。

这两类信息应该分开表达，但保持可追踪关系。

## 2. 稳定模型与变化模型

### 2.1 稳定模型

稳定模型描述当前有效的业务状态，包括：

- 领域；
- 能力；
- 业务概念；
- 规则和不变量；
- 状态及状态转换；
- 跨领域契约；
- 当前有效的业务场景。

它回答的是：

> 当前业务为什么能够这样运行？

### 2.2 变化模型

变化模型描述业务从旧状态演进到新状态的过程，包括：

- 业务目标；
- 项目或版本；
- 业务变更单元；
- 兼容和迁移策略；
- 灰度、降级和回滚方式；
- 验收证据；
- 上线后的业务结果。

它回答的是：

> 为什么要改变，具体改变什么，如何安全地改变，以及如何证明改变是正确的？

两者的关系是：

```text
当前业务模型
    ↓
业务目标提出变化
    ↓
形成业务变更集
    ↓
修改场景、能力、规则和契约
    ↓
通过测试、对账与数据证明变化正确
    ↓
形成新的当前业务模型
```

因此，不应该为每个版本复制一套完整业务模型，而应该维护：

```text
当前有效模型 + 每次版本的变化记录
```

## 3. 业务变更单元

需求、功能、项目和版本的颗粒度经常不一致：

- 一个需求可能包含多个业务变化；
- 一个业务变化可能跨越多个技术任务；
- 一个版本可能交付多项业务变化；
- 同一个业务变化也可能分阶段发布。

因此，需要一个比“需求”和“版本”更加稳定的中间对象，可以称为“业务变更单元”。

### 3.1 定义

业务变更单元表达一次拥有完整业务动机、影响范围和验证方式的业务变化。

它既能向前追溯业务目标，也能向后连接领域能力、契约、实现任务和验证证据，是横向业务推进和纵向能力演进的交叉点。

### 3.2 示例

```text
变更 ID：CHG-PAID-REVIVE
变更名称：支持玩家失败后付费复活

业务动机：
提高高意愿玩家的关卡完成率和付费转化。

影响场景：
- 新增：玩家失败后付费复活
- 修改：玩家挑战失败
- 修改：订单退款

领域变化：
- 可玩内容：扩展挑战状态恢复能力
- 商业化：扩展一次性权益能力
- 增长：复用实验归因能力
- 安全：复用支付风险判断能力

契约变化：
- 新增复活权益发放契约
- 新增复活权益消费结果通知

规则变化：
- 同一权益最多消费一次
- 已消费权益退款后的处理方式

数据变化：
- 新增权益类型
- 历史订单不需要迁移

验证方式：
- 领域回归
- 端到端场景验收
- 灰度数据观察
- 支付和权益对账

回滚策略：
关闭购买入口，但继续处理已创建订单和已发放权益。
```

### 3.3 建议字段

一个业务变更单元至少应表达：

1. 为什么变化；
2. 改变了哪些业务场景；
3. 涉及哪些领域；
4. 每个领域发生了什么性质的变化；
5. 改变了哪些领域规则和跨领域契约；
6. 对存量数据和进行中状态有什么影响；
7. 如何灰度、兼容、降级和回滚；
8. 如何证明业务和系统变化正确；
9. 谁对横向闭环、纵向能力和领域契约负责。

## 4. 对领域变化进行分类

仅仅记录“某领域受到影响”仍然过于模糊。为了判断设计和测试风险，需要进一步表达变化的性质。

| 变化类型 | 含义 | 主要风险 |
| --- | --- | --- |
| 复用 | 使用已有能力，不改变领域语义 | 场景组合风险 |
| 扩展 | 在原有模型上增加兼容性能力 | 历史回归风险 |
| 修改 | 改变已有规则、状态或业务语义 | 破坏历史逻辑 |
| 新建 | 现有领域无法合理承载，需要建立新能力或领域 | 边界设计风险 |
| 绕过 | 为快速交付临时绕开既有能力 | 能力分裂与长期债务 |
| 废弃 | 下线旧能力、规则或契约 | 存量数据和兼容风险 |

例如，“商业化需要支持付费复活”可以进一步拆解为：

- 订单能力：复用；
- 支付能力：复用；
- 权益能力：扩展；
- 退款规则：修改；
- 游戏状态恢复：在可玩内容领域新增能力。

一旦完成分类，设计评审、风险判断和测试范围就会更加明确。

### 4.1 对“绕过”的特殊处理

临时绕过不一定绝对错误。有时为了验证业务、处理紧急问题或满足关键时点，需要接受局部不完整。

但绕过必须显式记录：

- 为什么当前不能复用或扩展正式能力；
- 牺牲了哪些领域原则；
- 产生了什么风险；
- 影响哪些业务场景；
- 谁负责后续收敛；
- 在什么时间或条件下必须清理。

如果绕过没有退出条件，它实际上已经成为未被承认的新业务逻辑。

## 5. 补充跨领域业务不变量

领域内部存在不变量，但一些最重要的业务承诺无法归属于任何单一领域。

例如：

> 用户成功支付后，最终应该获得一次且仅一次的复活机会。

这条业务承诺横跨：

```text
支付 → 订单 → 权益发放 → 权益消费 → 游戏状态恢复
```

即使每个领域内部都没有违反自己的规则，整体仍然可能发生：

- 支付成功但没有发放权益；
- 同一次支付发放了两次权益；
- 权益消费成功但游戏状态没有恢复；
- 游戏状态已经恢复，但权益仍然可以再次消费；
- 发生退款后，权益和游戏状态不符合约定。

因此，不变量至少需要分成两类。

### 5.1 领域不变量

领域不变量由单个领域独立保证，例如：

```text
订单支付成功后不能再次支付
同一权益不能重复发放
一次性权益不能重复消费
游戏状态只能沿合法路径迁移
```

### 5.2 端到端不变量

端到端不变量由多个领域共同保证，例如：

```text
一次成功支付最终对应一次有效复活
任何一次复活都能追溯到合法权益
退款、权益和复活状态之间符合约定
```

端到端不变量不能只停留在口号层面，而应该进一步拆成各领域的明确责任：

```text
全局业务承诺
  → 商业化保证权益唯一发放
  → 可玩内容保证权益幂等消费
  → 跨领域契约保证同一业务标识贯穿全流程
  → 对账机制检查最终一致性
  → 异常任务负责重试或补偿
```

许多严重业务问题并不是发生在单一领域内部，而是发生在领域的交界处。因此，端到端不变量是横向业务正确性的核心表达。

## 6. 五层正确性

为了避免把“功能完成”误认为“业务正确”，可以将正确性分成五层。

| 正确性层次 | 回答的问题 | 主要保障手段 |
| --- | --- | --- |
| 领域正确性 | 单个领域内部是否自洽 | 规则校验、状态机测试、领域回归 |
| 契约正确性 | 领域之间是否正确理解彼此 | 契约测试、幂等验证、异常协议 |
| 场景正确性 | 用户或业务流程能否完整闭环 | 端到端验收、异常场景、业务对账 |
| 演进正确性 | 新旧版本、迁移、灰度和回滚过程中是否正确 | 兼容测试、迁移验证、回滚演练 |
| 观测正确性 | 异常发生后能否发现、定位和恢复 | 业务指标、链路标识、告警、补偿机制 |

### 6.1 观测也是业务正确性的一部分

一个系统即使能在理想路径下正常运行，但在生产异常发生后无法发现、定位、对账和补偿，也不能算真正自洽。

因此，一个完整业务场景还应表达：

- 使用什么业务标识串联全流程；
- 每个阶段产生什么可观测事实；
- 什么状态属于业务异常；
- 谁负责发现异常；
- 谁负责修复或补偿；
- 如何确认最终恢复正确。

## 7. 三个相互连接的闭环

横向、纵向和时间不应只形成三种静态视图，还应该形成三个持续运转的工作闭环。

### 7.1 横向交付闭环

```text
业务目标
→ 场景设计
→ 跨领域协作
→ 版本交付
→ 场景验收
→ 业务指标验证
```

它保证本次业务目标真正成立，而不是参与领域分别完成了自己的任务。

### 7.2 纵向能力闭环

```text
新需求
→ 识别领域变化
→ 复用或扩展能力
→ 更新领域规则
→ 建立回归保护
→ 被后续场景再次复用
```

它保证能力不是一次性交付，而是随着需求持续积累。

### 7.3 时间演进闭环

```text
旧业务状态
→ 变更设计
→ 兼容和迁移
→ 灰度发布
→ 稳定运行
→ 废弃旧逻辑
→ 形成新业务状态
```

它保证系统能够安全地从旧状态到达新状态，而不是只验证最终理想状态。

## 8. 嵌入业务产品研发过程

这套方法不应该额外建立一次孤立的“方法论评审”，而应该嵌入已有工作过程。

### 8.1 需求提出阶段

需要回答：

- 业务目标是什么；
- 哪些现有场景发生变化；
- 新场景的完整业务闭环是什么；
- 最终成功由什么业务结果衡量。

主要产物：业务目标、业务场景、成功指标。

### 8.2 方案设计阶段

需要回答：

- 涉及哪些领域和能力；
- 每个领域属于复用、扩展、修改、新建还是绕过；
- 哪些规则和契约发生变化；
- 是否产生新的端到端不变量；
- 对数据和进行中状态有什么影响。

主要产物：业务变更单元、领域变化、契约变化、不变量和决策记录。

### 8.3 开发测试阶段

应该根据变更内容推导：

- 领域回归范围；
- 契约测试范围；
- 端到端场景用例；
- 兼容、灰度和回滚验证；
- 对账与补偿验证。

主要产物：验证矩阵以及测试、演练和对账结果。

### 8.4 发布完成阶段

需要沉淀：

- 更新后的当前领域能力和业务场景；
- 本次版本的变化和重要决策；
- 验证结果；
- 业务指标是否达到预期；
- 临时绕过和后续收敛事项。

主要产物：新的当前业务模型、版本记录和业务结果复盘。

## 9. 横纵模型也是责任模型

横向和纵向对应两种不同的责任结构：

```text
横向：目标型、阶段性、跨团队
纵向：能力型、长期性、领域内
```

横向负责人倾向于尽快完成业务目标，纵向负责人倾向于维护领域一致性和长期演进。二者产生张力是正常现象。

真正需要建立的不是让一方压倒另一方，而是让取舍变得明确：

- 当前是在复用已有能力，还是扩展领域模型；
- 当前方案是正式能力，还是具有明确期限的临时方案；
- 临时方案产生了什么债务；
- 谁负责后续收敛；
- 什么情况下可以为了业务速度接受局部不完整；
- 什么变化因为不可逆或影响广泛，必须优先保证领域完整性。

横纵自洽不等于每一次变化都达到理论上的完美，而是每一次取舍都被识别、记录、验证和承担。

## 10. 双向可追踪链

整套方法可以进一步收敛为一条完整的追踪链：

```text
为什么变化
    业务目标
        ↓
改变什么业务
    业务场景
        ↓
依赖什么
    领域能力与契约
        ↓
必须保持什么正确
    领域不变量与端到端不变量
        ↓
本次如何变化
    业务变更单元
        ↓
如何证明正确
    测试、对账与业务指标
```

这条链条中的每一个对象，都应该能够：

- 向前解释“为什么存在”；
- 向后解释“如何实现”；
- 横向解释“与哪些业务共同协作”；
- 纵向解释“沉淀在哪个领域能力中”；
- 通过证据解释“如何证明是正确的”。

这套方法论真正需要建立的，不只是横纵关系图，而是：

> 从业务目标到业务场景、领域能力、业务规则、系统变化和验证证据的双向可追踪性。

有了这种可追踪性，横向业务推进和纵向能力沉淀才真正连接起来。

## 11. 下一步待探索的问题

当前模型仍有一些值得继续深入的问题：

1. 如何判断什么应该成为领域，什么只是一项能力？
2. 业务场景应该拆到什么颗粒度，才能既完整又可维护？
3. 如何从业务变更自动推导影响范围和测试范围？
4. 如何衡量一个领域是否真正形成了能力沉淀，而不是功能堆积？
5. 如何处理跨领域不变量的最终责任归属？
6. 如何把临时方案和领域债务纳入正常治理，而不是依赖个人记忆？
7. 什么情况下应该调整领域边界，而不是继续扩展原有能力？
8. 如何用工具生成横向、纵向、版本和验证视图，同时避免重复维护？
