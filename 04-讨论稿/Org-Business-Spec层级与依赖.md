# Org、Business、Spec 的层级与依赖

> 状态：讨论稿，尚未收敛  
> 目的：明确 Org、Business、Spec 分别表达什么，以及三者之间允许和禁止的依赖关系。当前结构已经基本成立，但仍存在尚未被识别的缺失，后续通过讨论继续校准。

## 1. 当前提出的层级

当前尝试建立以下层级：

```text
Org
→ Business
→ Spec
```

其中：

- `Org` 表达组织的行动能力；
- `Business` 表达业务本身，核心是横向、纵向和时间；
- `Spec` 偏向 SDD，负责把一次业务变化转化为可开发、可验证的描述。

这一层级不是简单的文档目录，也不是传统的组织命令链。它表达的是：

```text
Org 决定谁能够行动和改变业务
Business 描述当前业务世界及其目标变化
Spec 将已经明确的业务变化转化为开发过程
```

## 2. Org：行动与治理系统

Org 负责描述组织如何具备行动能力，包括：

- 组织结构；
- 角色；
- 人和智能体；
- 权限；
- 责任；
- 技能；
- 工具；
- 资源；
- 优先级；
- 决策与审批；
- 学习和能力沉淀。

Org 回答：

- 谁可以提出业务变化；
- 谁可以确认或拒绝业务变化；
- 谁可以修改哪些业务对象；
- 谁可以执行哪些开发和运行操作；
- 资源如何被分配；
- 组织如何从结果中学习和调整。

Org 可以操作、治理和观察 Business 与 Spec，但 Org 信息不应该被直接写入 Spec。

## 3. Business：横向、纵向和时间

Business 表达业务世界本身，不表达当前由哪个组织、团队、人或智能体负责实现。

Business 的核心结构是：

```text
横向：业务场景如何跨领域形成价值闭环
纵向：领域如何拥有事实、规则和能力
时间：业务如何从一个有效状态演进到另一个有效状态
```

Business 包含：

- 领域；
- 能力；
- 场景；
- 契约；
- 不变量；
- 当前有效业务状态；
- 业务变化；
- 生效条件；
- 历史业务演进。

Business 回答：

- 当前业务如何运行；
- 一个场景如何横向闭环；
- 每个领域纵向拥有什么能力；
- 什么业务规则必须成立；
- 业务准备从什么状态变成什么状态；
- 变化在什么时间和范围内有效。

业务语义不应该因为组织结构、执行人或实现技术发生变化。

例如，无论由哪个团队、人或智能体开发，下面的业务不变量都应该保持相同含义：

```text
一次支付最多产生一份复活权益
一次性权益最多完成一次复活
旧客户端继续运行旧业务场景
```

## 4. Business Change 属于 Business

业务变化是 Business 时间维度中的对象，而不是开发 Spec 本身。

业务变化表达：

```text
Business(t)
+ Business Change
→ Business(t+1)
```

它需要说明：

- Before / After；
- 横向业务场景发生什么变化；
- 纵向领域能力发生什么变化；
- 哪些契约和不变量发生变化；
- 变化何时、对谁生效；
- 什么业务结果必须成立。

业务变化不负责表达具体代码、系统设计和开发任务。

## 5. Spec：业务变化的开发表达

Spec 不描述完整的当前业务世界，而是引用一个已经在 Business 中明确的业务变化，并将其转化为开发过程。

```text
Business Change
→ Requirements
→ Design
→ Tasks
→ Verification
→ Implementation
```

Spec 回答：

- 为了实现该业务变化，产品行为需要发生什么变化；
- 当前系统需要修改什么；
- 数据、接口、配置和运行方式如何演进；
- 开发任务如何拆分；
- 如何验证实现满足业务变化；
- 如何部署、迁移、灰度和回滚。

因此：

```text
业务变化 ≠ 开发 Spec
```

两者的关系是：

```text
Business
├── 当前业务模型
└── Business Change CHG-xxx
          ↓
        Spec
        ├── Requirements
        ├── Design
        ├── Tasks
        └── Verification
```

## 6. Spec 不应该看到 Org

这是当前层级中的核心解耦原则。

Spec 不应该包含或依赖：

- 谁提出了需求；
- 谁批准了需求；
- 哪个组织或团队负责；
- 组织预算；
- 组织优先级；
- 人员绩效；
- 组织审批链；
- 由人还是智能体执行。

这些信息属于 Org 或外部任务编排系统。

Org 可以引用和管理 Spec：

```text
业务变化：CHG-PAID-REVIVE
执行 Spec：SPEC-PAID-REVIVE
执行角色：ROLE-xxx
执行主体：AGENT-xxx
```

但 Spec 内部不需要知道执行角色和执行主体是谁。

这可以形成单向关系：

```text
Org 可以观察和操作 Business 与 Spec
Spec 不能反向读取 Org
```

这样组织调整、人员变化和执行主体替换不会改变 Spec 的业务及开发语义。

## 7. “Spec 只能引用 Business”的精确含义

如果“引用”指业务语义来源，那么：

> Spec 的业务权威来源只能是 Business。

Spec 不能根据组织偏好或实现便利自行改变：

- 业务场景；
- 领域规则；
- 契约语义；
- 不变量；
- 业务生效条件；
- 必须达到的业务结果。

如果 Spec 发现业务变化存在歧义或冲突，它必须把问题返回 Business，而不是在设计或代码中自行作出业务决定。

但是，如果 Spec 指完整的 SDD 过程，那么它还需要了解当前实现上下文，否则无法形成可执行的 Design 和 Tasks。

当前实现上下文可能包括：

- 现有产品行为；
- 当前代码和模块结构；
- 已有系统和服务；
- 数据模型；
- 接口和事件；
- 部署环境；
- 技术约束和工程规范。

因此更准确的关系是：

```text
                  Business Change
                         ↓
当前实现 I(t) ─────→ Spec ─────→ 新实现 I(t+1)
```

可以概括为：

```text
Spec = Translate(Business Change, Current Implementation)
```

其中：

- Business 决定必须实现什么；
- 当前实现决定从什么现实状态开始改变；
- Spec 描述如何把当前实现变成能够承载目标业务的新实现；
- Org 不参与 Spec 的业务语义和实现设计。

所以当前规则不是“Spec 在任何意义上只能读取 Business”，而是：

> Spec 的业务依据只能来自 Business；Spec 可以读取实现上下文，但不能读取 Org，也不能让实现约束静默改变业务语义。

## 8. Org 不应该直接产生 Spec

如果 Org 直接产生 Spec，会形成：

```text
组织提出要求
→ 直接形成页面、接口和任务
```

这样会跳过 Business，导致开发任务缺少清晰业务语义。

正确关系应该是：

```text
Org
  提出、判断和授权业务变化
        ↓
Business
  将变化表达为横向、纵向和时间上的业务差异
        ↓
Spec
  将业务变化转化为 Requirements、Design、Tasks 和 Verification
```

Org 的意图只有先转化为 Business Change，才可以进入 Spec。

## 9. Spec 与真实实现不能等同

即使采用 SDD，也不能假设 Spec 已经形成就等于业务已经实现。

可能出现：

```text
Spec 已确认，但代码尚未实现
代码已经实现，但尚未部署
代码已经部署，但功能尚未生效
功能已经生效，但行为不符合 Spec
实现符合 Spec，但业务结果没有达到预期
```

因此完整运行链路仍然是：

```text
Org
→ Business
→ Spec
→ Implementation / Runtime
→ Evidence
```

Implementation 和 Evidence 可以不作为当前方法论的顶层知识层，但必须被承认为独立现实，不能被 Spec 替代。

## 10. 三层的稳定性差异

| 层级 | 主要沉淀 | 相对变化频率 |
| --- | --- | ---: |
| Org | 角色、权限、技能、工具和行动能力 | 较低 |
| Business | 横向场景、纵向领域、规则和业务状态 | 中等 |
| Spec | 一次具体变化的需求、设计、任务和验证 | 较高 |

这也可以被人和智能体理解为：

```text
Org 决定“你是谁、能做什么”
Business 告诉“当前业务世界是什么”
Spec 告诉“这次需要实现什么变化”
```

## 11. Spec 完成后的处理

Spec 完成后，不应将整份 Spec 合并进 Business。

```text
Spec 中的业务变化结果
→ 形成新的 Business 当前状态

Spec 中的实现设计
→ 保留在实现系统或技术文档

Spec 中的验证证据
→ 关联到业务变化、不变量和场景

Spec 本身
→ 作为一次开发变化的历史记录
```

Spec 不应该成为当前业务事实的永久容器。

## 12. 对现有“业务变更 Spec”的重新判断

现有“业务变更 Spec”同时包含了 Business、Spec 和部分 Org 信息。

在当前层级下，应该在逻辑上区分：

### 属于 Business

- 业务背景与业务问题；
- Before / After；
- 横向业务场景；
- 纵向领域影响；
- 契约和不变量；
- 业务生效条件。

### 属于 Spec

- 产品与系统行为；
- 技术设计；
- 数据迁移实现；
- 任务拆分；
- 测试和验证方案；
- 部署、灰度和回滚机制。

### 属于 Org 或编排系统

- 负责人；
- 审批人；
- 组织优先级；
- 资源；
- 执行人或智能体；
- 任务分配和组织流程状态。

当前不急于修改既有文档，先继续确认三层关系是否完整。

## 13. 当前阶段性结论

目前可以形成以下结构：

```text
Org
    行动主体、权限、能力和治理
        │
        │ 提出、授权和操作业务变化
        ▼
Business
    横向场景、纵向领域和时间演进
    当前业务模型 + Business Change
        │
        │ 提供唯一业务语义
        ▼
Spec
    Requirements → Design → Tasks → Verification
        │
        │ 驱动实现
        ▼
Implementation / Runtime
        │
        │ 产生证据
        └────────→ 校验 Business Change
```

核心依赖规则是：

1. Org 可以治理 Business 和 Spec；
2. Org 的意图必须先转化为 Business Change，不能直接成为 Spec；
3. Business 使用横向、纵向和时间表达业务本身；
4. Spec 的业务语义只能引用 Business；
5. 完整 SDD Spec 可以读取当前实现上下文；
6. Spec 不读取 Org；
7. 实现约束不能在 Spec 中静默改变 Business；
8. Spec、实现完成和 Business 生效是不同状态。

## 14. 尚未消除的缺失感

当前结构已经能够解释：

- 谁能够行动；
- 当前业务是什么；
- 一次业务变化如何进入开发；
- 开发如何产生实现和证据。

但这一结构仍然有某种尚未被准确识别的缺失感。

当前不为了追求形式完整而立刻增加新的层级或对象，也不预设缺少的一定是战略、环境、价值、知识、证据或运行层。

后续讨论应继续观察：

- 哪类现实问题仍然无法被 Org、Business、Spec 解释；
- 缺少的是一个独立层级，还是三层之间尚未定义的关系；
- 缺少的是静态对象，还是反馈、选择和演进机制；
- 当前的“层级”是否实际上混合了主体、业务世界和变化过程三种不同性质的事物。

在缺失被真实问题说明之前，暂不强行补全模型。
