# 业务模型之外的组织、实现与环境扩展

> 状态：讨论草稿  
> 目的：分析当前 `business-model/` 之外仍需表达的组织角色、专业技能、技术实现、外部环境和外部依赖，并明确它们与业务模型及业务变更 Spec 的关系。

## 1. 当前判断

当前问题不是简单缺少一个 `environment/` 目录，而是现有 `business-model/` 正在向“组织如何运行整个业务系统”扩展。

更合适的基本判断是：

> 保持 Business Model 的语义纯度，在它旁边建立 Organization、Realization、Ecosystem 等关联模型；Spec / Change 作为跨模型的变化单元，而不是 Business 的下一层。

现有方法论已经存在两个相互连接的层次：

```text
业务对象层：业务是什么、如何运行、如何变化
主体运行层：谁理解、谁决策、谁执行、谁验证、谁学习
```

业务对象层主要包含领域、能力、规则、契约、场景、版本和业务变化；主体运行层主要包含组织、角色、人、智能体、权限、行动、证据和学习。

因此，组织、角色、专业技能和技术实现不是新的业务领域对象，而是围绕业务模型运行的其他模型。

## 2. 当前目录仍缺少的业务对象

当前示例目录为：

```text
business-model/
├── README.md
├── glossary.md
├── domains/
├── capabilities/
├── scenarios/
├── contracts/
├── releases/
└── decisions/
```

这份目录来自方法论较早阶段，尚未覆盖后续已经明确的对象：

- `Invariant`：业务必须保持的正确性承诺；
- `Change / Spec`：对当前业务模型的一次具体变化；
- `Evidence`：证明不变量、契约、场景和变化正确的证据；
- `Effective Context`：模型或变化在什么时间、地区、租户、版本、用户范围和功能开关下有效。

需要明确区分：

```text
Change / Spec = 一次具体业务变化
Release       = 多个 Change 的发布集合
Decision      = 为什么选择某种方案
Evidence      = 凭什么相信结果正确
```

因此，仅修正当前业务模型时，目录至少可以扩展为：

```text
business-model/
├── README.md
├── glossary.md
├── domains/
├── capabilities/
├── scenarios/
├── contracts/
├── invariants/
├── changes/
├── releases/
├── decisions/
└── evidence/
```

## 3. 不应把所有缺失内容统一称为 Environment

“每一层都有环境”这个观察具有合理性，但更准确的表达是：

> 每一层都有自己的 Context、Constraint 和 Dependency；它们不都是 Environment。

几个概念需要分开：

| 概念 | 含义 | 示例 |
| --- | --- | --- |
| Context | 一项事实、规则或关系在什么条件下成立 | 地区、租户、用户群、时间、版本 |
| Environment | 组织无法直接控制、但会影响模型的外部条件 | 市场、监管、社会观念、技术趋势 |
| Constraint | 对行动形成限制的条件 | 法律要求、平台规则、成本上限 |
| Dependency | 完成业务结果所依赖的主体、能力或系统关系 | 支付渠道、短信供应商、外部风控 |
| Runtime Environment | 技术系统的运行位置和运行条件 | 开发、预发、生产、区域集群 |
| Resource | 可以分配或消耗的内部资源 | 人力、预算、算力、管理注意力 |

其中需要特别区分两种容易混淆的内容：

1. 地区、版本、灰度、租户和功能开关属于业务变化的生效上下文；
2. 市场、监管、合作伙伴、技术条件和成本结构属于外部环境。

如果把角色、技能、生产环境、支付平台和监管规则全部放进同一个 `environment/`，对象之间的语义差异会被掩盖，也难以进行责任、变化和影响分析。

## 4. 角色、专业领域和技能需要分开

判断一个概念是不是角色，不取决于职位名称，而取决于：

> 它是否在一个明确范围内拥有稳定的责任、权限和结果承诺。

建议采用以下术语：

| 类型 | 定义 | 示例 |
| --- | --- | --- |
| Role | 在特定范围内承担稳定责任和权限 | 商业化领域负责人、场景指标负责人、数据库变更审批人 |
| Discipline | 专业知识领域或专业方向 | 数据分析、Go 工程、数据库工程 |
| Skill | 可以学习、执行和验证的具体技能 | 留存分析、Go 性能诊断、在线表结构迁移 |
| Proficiency | 某个主体掌握某项技能的程度 | 熟悉、可独立执行、可评审、专家 |
| Actor | 实际行动主体 | 人、智能体、团队、外部合作方 |
| Assignment | Actor 在一段时间和范围内承担 Role 的关系 | 某人担任商业化领域负责人 |

由此可以判断：

- “数据分析师”通常首先是职位或专业方向；
- 数据分析师在具体业务变化中可能承担“成功指标定义者”“实验分析负责人”等角色；
- “Go 研发工程师”通常是专业方向加技能集合，不天然是方法论中的责任角色；
- “数据库工程师”也是如此；只有承担数据库变更审批、数据恢复或数据安全责任时，才表现为 Role；
- “领域专家”可以成为 Role，但必须写清楚他对哪些业务语义、规则、评审和决策负责。

当前方法论中的 `Capability` 已经表示领域基于业务事实、状态和规则，对外提供的业务行为或判断。因此，人的基础知识不应继续称为 Capability，建议使用 `Discipline`、`Skill`、`Competency` 和 `Proficiency`。

角色与主体之间应保持解耦：

```text
稳定角色
    ↓ assigned_to
当前的人 / 智能体 / 团队
```

这样，人员变化不会导致责任边界和组织记忆一起消失。

## 5. Org → Business → Spec 不适合成为严格层级

Org、Business 和 Spec 不是简单的父子包含关系：

- 一个组织单元可能维护多个业务域；
- 一个业务域可能由多个组织单元共同维护；
- 一个 Spec 可能同时改变多个业务域、组织责任、技术系统和外部依赖；
- 组织会重组，但业务事实未必随之改变；
- 技术系统会被替换，但业务能力、契约和不变量可能保持稳定。

更适合的结构是多个相互关联的模型：

```text
外部环境 / 生态
        │ 约束、驱动、依赖
        ▼
业务模型 ─────── realized_by ────── 技术实现模型
   │                                      │
   │ owned_by / accountable_by            │ operated_by
   ▼                                      ▼
组织运行模型 ───── requires_skill ─── 人 / 智能体
        ▲
        │ changes
   Change / Spec
        │
        ▼
   Evidence / Feedback
```

各个模型内部可以存在层级：

```text
Org Unit 包含下级 Org Unit
Domain 拥有 Capability
System 包含 Service / Component
Release 包含 Change
```

模型之间应该使用稳定引用和关系连接，不应依靠目录嵌套表达所有权和影响关系。

## 6. 业务域与技术设计实现的关联

不建议把技术设计实现继续称为“技术域”，因为当前 `Domain` 已被定义为业务责任和业务事实的所有权边界。再次使用“域”容易混淆业务语义与技术模块。

更合适的两个名称是：

```text
Business Model     业务是什么、为什么成立
Realization Model  业务由什么系统、数据和接口实现
```

两者可以通过以下关系连接：

```text
Business Capability  → realized_by     → Service / System
Business Contract    → implemented_by  → API / Event / Protocol
Business Invariant   → enforced_by     → Code / Schema / Test / Monitor
Business Scenario    → executed_through → System Flow
Business Domain      → stewarded_by    → Role / Org Unit
```

这种分离具有两个直接价值：

1. 技术实现被替换时，不需要重新定义稳定的业务模型；
2. 业务语义发生变化时，可以反向找到受影响的系统、数据、接口和验证证据。

## 7. 外部依赖的表达方式

外部依赖值得进入可维护的源事实，但它本质上是一种关系，而不只是环境说明。

外部系统和依赖关系应分别表达：

```text
External System：Apple IAP
Dependency：支付能力依赖 Apple IAP 完成 iOS 支付
```

因为多个业务能力可能以不同的业务语义、关键程度和失败方式依赖同一个外部系统。

一项重要外部依赖至少可以记录：

```text
依赖 ID：DEP-APPSTORE-PAYMENT
依赖方：CAP-PAYMENT
被依赖对象：EXT-APPLE-IAP
依赖语义：完成 iOS 用户支付确认
关键性：核心
外部契约：苹果支付协议
失败影响：无法创建新的支付结果
降级方式：禁止发起新支付，保留订单查询
一致性要求：支付结果必须可对账
负责人：商业化领域负责人
复核条件：协议、费率或区域政策变化
证据：对账、可用性指标、故障演练
```

并非所有依赖都需要立即建立独立文件。满足以下任一条件时，适合成为独立源对象：

- 被多个领域、能力、场景或系统引用；
- 拥有独立负责人、契约或生命周期；
- 变化后需要进行影响分析；
- 失败会改变业务结果或需要明确降级；
- 需要持续观测、复核和治理。

简单且只在一个对象内部使用的依赖，可以先作为该对象的属性存在。

## 8. Source 的两种含义

需要区分“源对象”和“来源信息”。

### 8.1 Source of Truth

指可以直接编辑和长期维护的权威对象及其关系，例如 Domain、Capability、Contract、Scenario、Invariant 和 Change。派生视图不应成为另一份重复维护的数据。

组织角色、技术系统和外部依赖如果需要被复用、追踪和独立演进，也可以成为各自模型中的源对象。

### 8.2 Provenance

指一项判断或事实来自哪里，例如：

- 业务决策；
- 用户研究；
- 外部协议；
- 法律法规；
- 代码和数据库现状；
- 指标、实验和故障证据。

因此，`source` 更适合作为每个对象的来源元信息，而不建议建立一个 `source/` 目录，把 Org、Skill、Dependency 和 Environment 混合存放。

## 9. 推荐的整体目录

如果方法论继续从 Business Model 扩展为业务操作系统，可以将总根目录提升为更中性的 `model/` 或 `operating-model/`：

```text
model/
├── README.md
├── glossary.md
├── business/
│   ├── domains/
│   ├── capabilities/
│   ├── scenarios/
│   ├── contracts/
│   └── invariants/
├── organization/
│   ├── org-units/
│   ├── roles/
│   ├── disciplines/
│   ├── skills/
│   ├── actors/
│   └── assignments/
├── realization/
│   ├── systems/
│   ├── services/
│   ├── data/
│   ├── interfaces/
│   └── mappings/
├── ecosystem/
│   ├── external-actors/
│   ├── external-systems/
│   ├── dependencies/
│   ├── constraints/
│   └── environmental-factors/
├── changes/
├── releases/
├── decisions/
└── evidence/
```

其中：

- `business/` 保存稳定业务事实和业务语义；
- `organization/` 保存组织、角色、技能、权限和主体绑定；
- `realization/` 保存系统、服务、数据和接口，以及它们对业务的实现关系；
- `ecosystem/` 保存外部主体、系统、约束、环境因素和依赖关系；
- `changes/` 保存跨模型变化；
- `releases/` 将多个变化组织为一次发布；
- `decisions/` 保存重要取舍及其背景；
- `evidence/` 保存验证业务、实现和变化正确性的证据。

## 10. 各层共有的上下文元信息

“每一层都有环境”不必通过在每层都建立一个 `environment/` 目录解决。更适合的方式是为重要源对象提供统一的上下文元信息：

```text
id
type
owner
scope
effective_when
depends_on
constraints
source
evidence
status
valid_from
valid_to
review_at
```

这些元信息共同回答：

- 这是什么对象；
- 谁对它负责；
- 它在什么范围和条件下有效；
- 它依赖什么、受什么约束；
- 这个判断来自哪里；
- 有什么证据支持；
- 何时需要重新检查或废弃。

不是所有字段都必须出现在所有对象中，但应保持含义一致。

## 11. 方法论边界与推进建议

上述内容并不说明当前 Business Model V1 是错误或不完整的。Business Model V1 的目标仍然可以保持为：

- 描述当前业务状态；
- 描述跨领域业务场景；
- 描述领域能力和契约；
- 描述业务变化；
- 证明变化后的业务仍然正确。

组织、技术实现和外部生态属于围绕核心业务模型的第二圈模型。推进时可以分两步：

### 第一步：补齐 Business Model 核心

- 增加 Invariant、Change 和 Evidence；
- 明确 Release 只是 Change 的集合；
- 为业务对象增加 owner、scope、effective_when、source 和 evidence 等最低元信息；
- 用一个真实业务变化验证闭环。

### 第二步：按真实阻力增加关联模型

- 当责任与人员变化成为阻力时，建立 Organization Model；
- 当业务影响无法映射到代码和系统时，建立 Realization Model；
- 当外部系统、监管和合作方变化频繁影响业务时，建立 Ecosystem Model；
- 将 Change / Spec 提升为可以修改多个模型的统一变化单元。

这样既不会把所有现实提前压入 Business Model，也为后续形成完整的业务操作系统保留了结构。

## 12. 最终收敛

整个扩展可以收敛为一句话：

> Business 描述业务事实，Organization 描述谁负责，Realization 描述如何实现，Ecosystem 描述外部世界，Context 描述何时成立，Spec 描述它们如何一起变化，Evidence 描述为什么相信变化是正确的。

当前想到的角色、技能、技术设计和外部依赖，不应该被统一压入 `environment`，也不应直接污染已经收敛的 Business Model。它们更适合作为围绕业务核心的组织模型、实现模型和生态模型，并通过稳定关系、变化单元和证据与业务模型连接。
